What I'd change:
Task 2 — City mapping is fragile and will rot. You're asking for "at least the top 50 cities by population" hardcoded into a curated list. This works short-term but creates a maintenance problem. A better approach: use the ZIP-to-state lookup as primary (it's already built), state abbreviation/name regex as secondary, and city mapping as a fallback with just the top ~20 cities that are unambiguous. "Portland" maps to both OR and ME. "Kansas City" maps to both KS and MO. "Vancouver" maps to WA. The spec should explicitly call out ambiguous cities and say how to handle them — either skip them from the mapping entirely, or require a second signal (state abbreviation or ZIP) to disambiguate.
Task 3 — The RED/NO-GO override needs a guard rail. Right now the rule engine presumably scores quotes on multiple factors. If a cap violation auto-forces "NO-GO — DOC FEE EXCEEDS STATE CAP," that's correct behavior, but the spec should state that this is a hard override — meaning even if every other signal in the quote looks clean, a cap violation alone is enough for NO-GO. Without saying this explicitly, the agent might implement it as just another weighted signal that could get averaged away.
Task 4 — Missing a key detail about prompt token budget. You're injecting a STATE_FEE_REFERENCE block into the system prompt. The state JSON entries are fairly verbose (cap info, tax rates, sources, special notes, typical range). That's fine for one state, but the spec should note that you're injecting only the detected state's block, not the entire 51-state JSON. This seems obvious to you and me, but a Replit agent could misinterpret "inject state context" and dump the whole file. Add one sentence: "Inject only the single detected state's data object, not the full dataset."
Task 5 — The statute citation instruction is underspecified. You say "from specialNotes or the state name" but not all specialNotes entries contain statute citations. Looking at the JSON, some states have notes about CPI adjustments or TAVT rates rather than statute numbers. The spec should say: "If specialNotes contains a statute reference, cite it. If not, cite the state name and cap amount without a statute number. Never fabricate a statute citation." This prevents the LLM from hallucinating legal codes.
Task 6 — Good, but add one thing. When the state is unknown, you correctly add a missingInfo entry. But also consider: if the user did mention a city that's ambiguous (like Portland), the missingInfo message should acknowledge the ambiguity rather than just generically asking for a ZIP. Something like: "We found a reference to Portland, which could be Oregon or Maine. What's the dealership's ZIP code?"
One missing task: There's no mention of logging or analytics for state detection. You'll want to know: what percentage of quotes successfully detect a state? Which detection method triggered (ZIP vs. abbreviation vs. city)? How often does detection fail? Even a simple console.log or a counter in your existing analytics would help you understand whether users are providing enough context for state detection to work. If detection fails 60% of the time, the entire feature underperforms silently.
