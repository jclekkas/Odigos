#110 - Data Quality Hardening: DLQ, Content-Hash Dedup, Financial Bounds

Implement this task exactly as specified. Make the smallest clean change set that fully completes the work without unrelated refactors.

What & Why
The warehouse ingestion pipeline currently scores ~5/10 on data quality. Three specific gaps degrade downstream data usefulness:
1. warehouse write failures are silently dropped
2. identical documents can be stored as separate submissions
3. extracted financial figures are never checked for plausibility beyond the existing doc-fee cap logic

Done looks like
- Identical document submissions (same content, same or different users) are detected via SHA-256 hash and the warehouse listing is marked as a duplicate rather than creating a new novel listing
- When a warehouse write fails, the failure is recorded in a failed_warehouse_writes table with the submission ID, serialized payload, error message, timestamp, and retry count
- A retry mechanism attempts failed warehouse writes up to 3 times before writing to the DLQ
- Extracted financial figures (vehicle price, trade-in value, fees, add-on costs) are validated against configurable sanity bounds before being written to the warehouse
- Out-of-bounds values are flagged in the listing row rather than rejected
- Existing analysis flow continues to work
- Existing doc-fee cap rule engine logic is unchanged

Out of scope
- External queue infrastructure (SQS, RabbitMQ, etc.)
- Admin UI for replaying DLQ entries
- Fuzzy duplicate detection
- Changing the existing doc-fee cap rule engine logic
- Unrelated refactors

Required implementation

1) Content-hash dedup
- At ingest time, compute a SHA-256 hash of the raw submission text
- Use a minimal deterministic normalization before hashing:
  - convert CRLF to LF
  - trim outer whitespace
  - do not lowercase
  - do not aggressively collapse internal whitespace
- Store the hash on dealer_submissions as content_hash
- Also store content_hash on the warehouse listing row so dedup can be checked cheaply in the warehouse path
- In warehouseWriter, before inserting a new listing, check whether an existing non-duplicate listing already exists with the same content_hash
- If one exists:
  - insert the current listing row as a duplicate record rather than as a novel listing
  - set is_duplicate = true
  - set duplicate_of_listing_id = original listing id
  - still link the row to the current submission so submission history is preserved
- If no original exists, insert normally with is_duplicate = false
- Do not make dealer_submissions.content_hash unique; multiple submissions may share it
- Do not skip writing the duplicate row entirely; preserve lineage

2) In-DB dead-letter queue
- Add a failed_warehouse_writes table in the app DB schema
- Minimum fields:
  - id
  - submission_id
  - payload (JSON/JSONB)
  - error_message
  - attempt_count
  - failed_at timestamp
- Wrap the warehouse write entrypoint in a retry loop
- Retry up to 3 total attempts with lightweight exponential backoff
- Use simple in-process backoff only (setTimeout is fine)
- If all attempts fail:
  - insert a row into failed_warehouse_writes
  - include submission ID
  - include a safely serialized warehouse payload
  - include error message
  - include final attempt count
- Do not silently swallow the failure anymore
- Be careful not to store raw document text in the DLQ payload if the payload currently includes it; prefer the warehouse payload / extracted structured fields only

3) Financial sanity bounds
- Add a configurable bounds map for common extracted numeric fields
- Validate these before the warehouse listing insert
- Do not reject the write if values are out of bounds
- Instead, write the row and store structured anomaly flags in a sanity_flags JSONB column on core.listings
- sanity_flags should be an array of objects, not just booleans
- Recommended fields to validate if present:
  - vehicle price
  - trade-in value
  - doc fee
  - dealer add-on cost
  - total fees
  - optionally down payment / monthly payment if already cleanly available
- Use reasonable configurable defaults such as:
  - vehiclePrice: 500 to 500000
  - tradeInValue: 0 to 500000
  - docFee: 0 to 10000
  - dealerAddonCost: 0 to 100000
  - totalFees: 0 to 10000
- For each out-of-bounds value, store a structured flag like:
  - field
  - value
  - min and/or max
  - reason: below_min / above_max / not_finite
- Missing fields should not generate flags
- Existing doc-fee cap rule logic stays unchanged; this is separate data-quality validation, not rule scoring

Behavior requirements / edge cases
- Same content from different users should still count as duplicate
- Exact-match dedup only; do not implement fuzzy matching
- Dedup should check for an existing original warehouse listing, not just any submission hash
- If an earlier submission failed to write to the warehouse and no original listing exists yet, a later identical submission may become the first successful original; that is acceptable
- Missing numeric values should not be flagged
- NaN / Infinity should be flagged as not_finite if they make it to validation
- Preserve existing normal write behavior for valid non-duplicate submissions

Testing requirements

- out-of-bounds financial values still insert the listing and populate sanity_flags
- existing doc-fee cap logic remains unchanged
