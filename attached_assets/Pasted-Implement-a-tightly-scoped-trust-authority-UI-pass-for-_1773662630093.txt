1. Masthead Header
2. Engine Box Analyzer

REQUIRED TASK 1 — MASTHEAD HEADER

Update the shared header in client/src/components/SiteHeader.tsx.

Problem:
The current header/logo treatment is too weak. The logo feels too small, the top of the page does not feel anchored, and the header reads more like a generic app navbar than a serious brand masthead.

Requirements:
- Increase the visual authority of the logo materially, not trivially
- Increase logo scale approximately 20–35% from current treatment
- Rebalance header spacing so the larger logo does not feel cramped
- Increase vertical padding in the header container
- Ensure the logo aligns exactly with the main page content container/grid
- Slightly strengthen the header’s bottom border/backdrop separation from page content
- Keep the header calm and restrained
- Preserve existing navigation behavior and layout logic
- Preserve responsive behavior, but ensure the logo does not become tiny or cramped on mobile
- The end result should make the brand feel intentionally placed and more premium

Do NOT:
- invent a new brand mark
- add decorative effects
- radically change nav structure

Success criteria for Task 1:
- the logo is noticeably larger
- the header feels more stable and deliberate
- the top-left brand presence clearly anchors the page

REQUIRED TASK 2 — ENGINE BOX ANALYZER

Update the analyzer section in client/src/pages/home.tsx (analyzer section).

Problem:
The analyzer is the product, but it does not assert itself enough as the primary functional object on the page.

Requirements:
- Give the analyzer container a stronger but restrained visual presence
- Use slightly crisper border/surface treatment so it feels more intentional than surrounding content
- Increase the heading size/weight above the analyzer slightly so the tool feels more important
- Tighten spacing and grouping between the textarea and submit button so the CTA hierarchy is clearer
- Keep the analyzer simple, clean, and serious
- It should feel like the engine of the page, not just another card

Do NOT:
- change form logic
- change field names
- change submission behavior
- add extra steps or friction
- make it look like a flashy SaaS widget

Success criteria for Task 2:
- the analyzer is visually the focal functional object
- its container feels more deliberate than surrounding sections
- the CTA hierarchy is clearer

REQUIRED TASK 3 — PROOF-OF-OUTPUT PREVIEW

Update client/src/pages/landing.tsx.

Add a static example output preview block on the landing page.
Place it in a sensible location near conversion-oriented content, such as before or after pricing.

Problem:
Users currently have to imagine what Odigos returns. This creates trust friction.

Requirements:
- Add a short static sample-output block showing what Odigos analysis looks like
- Include:
  - a verdict label/chip such as “RED / NO-GO”
  - 2–3 concise detected issues
  - one short realistic explanatory line
- This should resemble an actual excerpt of Odigos output
- Keep it editorial and product-real
- It should look like a sample result excerpt, not a dashboard, not a browser mockup, not a decorative card
- Keep it compact and credible

Example content direction:
Verdict: RED / NO-GO
Issues detected:
- Missing out-the-door price
- Dealer add-ons presented as mandatory
- Monthly payment framing without total cost
Short note:
“This quote leaves out the full price and uses common dealership framing that makes comparison harder.”

Do NOT:
- add charts
- add fake analytics
- add browser chrome
- add generic SaaS UI
- add decorative icons unless the current design system already uses them very sparingly

Success criteria for Task 3:
- a first-time visitor can immediately understand what the product output looks like
- the preview block feels realistic and restrained
