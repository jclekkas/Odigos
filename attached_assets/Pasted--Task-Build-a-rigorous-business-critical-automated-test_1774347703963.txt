# Task: Build a rigorous, business-critical automated test suite for Odigos

You are working on the Odigos codebase. I want a production-grade automated test suite that provides real confidence across the product — including backend logic, API behavior, UI/pages, user interactions, and core UX flows.

This is NOT a superficial test pass. This is a foundational system for engineering confidence.

---

## PRIMARY GOAL

Build a comprehensive but intelligently prioritized test suite covering:

1. core business logic
2. API endpoints
3. upload and extraction boundaries
4. page rendering and routing
5. critical buttons and interactions
6. core UX flows (end-to-end)
7. accessibility-critical behavior

The goal is **maximum confidence per line of test code**, not maximum number of tests.

---

## CONTEXT

Odigos is a trust-sensitive consumer protection tool for car buyers.

Users:
- paste or upload dealer quotes/messages
- receive a GO / NO-GO verdict + issues
- optionally pay for a full review

Regressions in:
- analysis flow
- upload handling
- API contracts
- pricing boundaries
- CTAs and conversion paths

are unacceptable.

---

## PRIORITIZATION RULE (CRITICAL)

Do NOT treat all UI elements equally.

Prioritize test coverage based on business impact:

### 🔴 Tier 1 — Must never break (highest priority)
- analyze flow (paste + upload → result)
- API contracts
- issue detection / rule logic
- free vs paid boundary
- primary CTAs (analyze, upgrade)
- upload validation and extraction behavior

### 🟡 Tier 2 — Should not break
- landing page and key SEO/article pages
- navigation and routing
- article → analyzer conversion flows
- key UX interactions (forms, accordions, copy buttons)
- trust-critical UI content presence

### 🟢 Tier 3 — Low priority
- decorative UI
- styling-only elements
- low-impact buttons

Focus heavily on Tier 1 and Tier 2.
Do NOT waste time exhaustively testing low-impact UI.

---

## TESTING STRATEGY

Implement a layered approach using appropriate tools:

- Unit tests → pure logic
- Integration tests → API routes
- Component/Page tests → UI behavior
- End-to-End tests → real user journeys

Preferred stack (if compatible with repo):
- Vitest (unit + integration)
- Testing Library (React behavior)
- Playwright (E2E)

Use the lightest credible setup that fits the repo.

---

## WHAT TO DO

### 1. Inspect the codebase first

Map:
- all routes/pages
- main CTAs and buttons
- forms and inputs
- upload flow
- analysis pipeline
- free vs paid boundaries
- result states and error states
- navigation structure

---

### 2. Set up test infrastructure

Add a clean, minimal setup for:
- unit tests
- component tests
- API tests
- E2E tests

Keep configuration simple and aligned with the repo.

---

### 3. Unit tests (core logic)

Add strong unit tests for deterministic logic:

- issue detection / rule engine
- severity classification
- scoring / verdict logic
- data transformation helpers
- state fee / cap lookup logic
- any logic affecting user-visible outputs

Focus on correctness and edge cases.

---

### 4. API / backend tests

Add integration-style tests for key routes:

Test:
- happy path response shape
- invalid input handling
- missing input handling
- malformed payloads
- upload-related errors
- correct status codes and messages

Mock:
- external AI APIs
- any external services

Do NOT call real APIs.

---

### 5. Upload & extraction boundary tests

Cover high-risk input scenarios:

- unsupported MIME types
- oversized files
- missing files
- no readable text
- extraction failure handling

Ensure:
- correct errors
- no crashes
- predictable behavior

---

### 6. Page & component tests (UI coverage)

For Tier 1 and Tier 2 pages:

Test:
- page renders without crashing
- critical headings and sections exist
- primary CTAs are present
- links/buttons point to correct destinations
- major sections required for UX are present

Pages to prioritize:
- landing/home
- analyze page
- key article/SEO pages
- results/review UI
- upload-related UI

Avoid brittle snapshot tests.

---

### 7. Button & interaction testing

Test important interactions across the app:

- primary CTAs (analyze, upgrade)
- submit/analyze actions
- upload triggers
- copy-to-clipboard behavior
- FAQ/accordion expand/collapse
- navigation actions
- retry/error recovery flows

Verify:
- element renders
- correct enabled/disabled state
- click + keyboard interaction works
- correct behavior is triggered

Do NOT exhaustively test trivial/decorative buttons.

---

### 8. Accessibility-critical tests

Add meaningful accessibility checks:

- buttons/links have accessible names
- form inputs have labels
- keyboard interaction works for key flows
- aria-expanded / aria-controls on accordions
- presence of core landmarks (main, etc.)
- skip link behavior if present

Do NOT over-engineer full accessibility auditing.

---

### 9. End-to-End (E2E) tests

Use Playwright (or best-fit tool) to test real user journeys:

At minimum:

- user lands → navigates to analyzer
- user submits pasted input → sees result or validation error
- user attempts invalid input → sees correct error
- upload flow works through UI (mock external dependencies)
- primary CTAs route correctly
- article → analyzer conversion path works
- pricing boundary behavior (if safely testable)

Mock external APIs where needed.

---

### 10. Minimal refactors only if necessary

If parts of the code are hard to test:

- extract pure functions where appropriate
- make small, safe refactors
- do NOT redesign the app
- do NOT change behavior unless fixing a real bug

If a bug is found, explain it clearly.

---

## NON-NEGOTIABLE CONSTRAINTS

- Do NOT redesign UI or product flows
- Do NOT introduce unnecessary frameworks
- Do NOT add brittle snapshot-heavy tests
- Do NOT call real external APIs
- Do NOT over-test styling
- Do NOT ignore UX-level coverage

- validate backend routes and contracts
- include real E2E journeys
- catch meaningful regressions
- be maintainable and extensible
