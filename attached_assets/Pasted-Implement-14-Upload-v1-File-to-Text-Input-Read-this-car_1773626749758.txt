Implement #14 — Upload v1: File-to-Text Input

Read this carefully: this is a tightly scoped change. Preserve the current Odigos product and workflow. Do not expand scope.

PRIMARY GOAL
Add a secondary input option on the analyzer page so users can upload a screenshot or PDF of a dealer quote instead of pasting text. The uploaded file must be converted to text on the server, then populate the existing dealerText field and continue through the current analysis flow unchanged.

This is a friction-reduction feature only.
It is NOT a redesign.
It is NOT a new workflow.
It is NOT a new analysis system.

DONE LOOKS LIKE
- A small “or upload a screenshot or PDF” affordance appears below the paste textarea
- Accepted file types: PNG, JPG, JPEG, WEBP, PDF
- All other types show a short rejection message
- Files over 10 MB are rejected with a short message before any server call
- On upload, a brief loading indicator appears
- Once extraction completes, the existing textarea is filled with the extracted text
- The user reviews the text and submits normally using the existing analysis path
- If extraction yields too little text (< 20 chars), show a calm prompt to paste manually or try a clearer image
- All errors are non-crashing; the page remains usable and the textarea remains editable
- Existing paste-text flow remains completely unchanged
- No new routes/pages/navigation beyond the single API endpoint needed for extraction
- Build passes with no TypeScript errors

IMPLEMENTATION TASKS

1) Install server dependencies
Add:
- multer
- pdf-parse

Add the appropriate TypeScript types as dev dependencies if needed.

No frontend dependencies.

2) Create server/extractText.ts
Create a narrowly scoped helper exporting one async function:

extractTextFromFile(buffer, mimetype)

Behavior:
- For PNG/JPG/JPEG/WEBP:
  - Call the existing OpenAI gpt-4o instance
  - Send the image as base64 in a vision request
  - Prompt it to return only the visible text, with no commentary or formatting beyond plain extracted text
- For PDFs:
  - First use pdf-parse to extract embedded text
  - If the extracted result is under 50 characters, fall back to the GPT-4o vision path using the raw buffer as base64
- Throw descriptive errors on failure

Important:
- Keep this helper narrowly scoped
- Reuse existing OpenAI server setup if one already exists
- Do not create a parallel AI client architecture
- Do not write files to disk

3) Add POST /api/extract-text in server/routes.ts
Use route-level multer middleware with:
- memoryStorage()
- fileSize limit: 10 MB

Server requirements:
- Validate MIME type against the allowed list
- Reject unsupported types with HTTP 400 and a short user-facing message
- Call extractTextFromFile and return:
  { text }
- If extraction fails or returns effectively empty text, return HTTP 422 with a calm user-facing message
- Never write uploaded files to disk
- Keep endpoint minimal and self-contained

4) Add upload UI + handler in client/src/pages/home.tsx
Inside the existing “Paste Dealer Communication” card, below the textarea:
- Add a hidden input type="file" using useRef
- Add a small text-style button that triggers the file input
- Add local state for:
  - uploadLoading
  - uploadError

Client behavior:
- On file select, validate type and size client-side before any server call
- Use raw fetch to POST multipart/form-data to /api/extract-text
- Do not use apiRequest for this since the request is not JSON
- On success:
  - call form.setValue("dealerText", extractedText)
  - clear any uploadError
- On failure:
  - set a plain uploadError message beneath the textarea
- Show a small Loader2 spinner during upload
- Do not add new components
- Do not use modals
- Do not make layout changes outside what is needed for the upload affordance
