Harden PII deletion job audit surface + semantics
What & Why
Task #91 implemented the core 90-day PII cleanup job well, but it is not fully complete from a privacy/compliance hardening standpoint yet.
The current implementation correctly redacts expired PII in the known core tables, but there are still important follow-up risks:
the audit surface may be incomplete
retentionExpiresAt is being reused with a second meaning
logs do not clearly expose eligible-vs-cleared row counts
the exact 90-day boundary operator must be explicitly documented
downstream warehouse/materialized view safety after redaction should be verified
This task is a hardening pass, not a redesign.
Goal
Make the PII cleanup implementation fully explicit, defensible, and operationally safe by tightening audit coverage, field semantics, observability, and downstream guarantees.
Done looks like
server/jobs/piiCleanup.ts contains a clear audited inventory of all checked PII surfaces, not just the tables touched
The code explicitly documents why each table/source is:
redacted
hard-deleted
or intentionally excluded
The exact retention boundary rule is documented in code and tests using a single explicit operator (< cutoff or <= cutoff)
Structured logs include:
scanned row count
eligible row count
cleared row count
skipped/already-cleared row count where relevant
retentionExpiresAt is no longer left semantically ambiguous
If that field must remain reused temporarily, the code clearly documents that reuse and prevents future misuse
Downstream warehouse behavior is verified so redacted rows do not break materialized views, warehouse refreshes, or analytics queries
Any known non-table PII surfaces checked during the audit are explicitly listed and dispositioned (for example logs, temp upload flow, analysis artifacts, Stripe metadata if relevant)
Acceptance report clearly states:
every surface checked
every surface touched
every intentional exclusion
any remaining technical debt
Out of scope
Full privacy portal
New user-facing features
New retention policies
Re-architecting the warehouse
Adding new product tables unless absolutely required
Broad logging refactors unrelated to PII retention
Full legal review

2. Clarify field semantics for retentionExpiresAt
The current implementation reuses retentionExpiresAt as a post-clear marker for raw.user_analyses.
That is semantically dangerous because:
expiry timestamp and cleared-at timestamp are not the same concept
future code could misread this field
Do one of the following:
Preferred
If there is any safe way to avoid overloading the meaning without violating task constraints, do so.
If reuse must remain
Keep the current schema, but add:
an explicit code comment wherever the field is written
an explicit comment in the job header block
a guard against treating the field as an active expiry indicator after the row is cleared
Also search the codebase for any reads of retentionExpiresAt and verify they will not misbehave after cleanup.
Report all usages found.
