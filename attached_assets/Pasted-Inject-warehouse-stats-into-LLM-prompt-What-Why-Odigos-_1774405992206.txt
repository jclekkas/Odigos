Inject warehouse stats into LLM prompt
What & Why
Odigos already injects state fee reference data into the GPT-4o system prompt inside server/routes.ts before the OpenAI call. However, it does not currently inject warehouse-derived market context such as state-level average doc fee and state deal volume. That means the AI summary and reasoning can say a fee “seems high” without grounding that claim in Odigos platform data.
This task extends the existing prompt-injection flow so the model can reference warehouse stats when available, without changing behavior when data is unavailable. In the current architecture, dealer name is not available until after the LLM extracts it, so dealer-level historical averages may need to remain a post-LLM enrichment unless you can prove there is already a safe pre-LLM source for dealer identity.
Done looks like
/api/analyze fetches warehouse market context before the GPT-4o call, after state detection and before openai.chat.completions.create(...)
The system prompt includes a MARKET_INTELLIGENCE block only when market context contains at least one usable statistic
The block gives the model plain-English figures it can cite directly, such as:
state code
number of analyzed deals in that state
state average doc fee, when present
If the warehouse lookup fails, times out, or returns insufficient data, the market block is omitted and the LLM path behaves exactly as before
The API response includes the same marketContext object used for prompt injection, so the frontend can render it
client/src/pages/home.tsx renders a new or updated “How This Compares” card only when marketContext is present
That card uses short narrative phrases, not label/value rows
If dealer name is only available post-LLM, dealer average score/count may be enriched after the LLM call and rendered in the UI, but do not pretend it was available to the first model pass
The existing state-fee injection, free preview flow, paid unlock flow, and response schema validation continue to work
Out of scope
Changing the existing state fee prompt injection logic except to append warehouse-based context
Fuzzy dealer matching or any new dealer name normalization logic
Fine-tuning or retraining the model
Any redesign of the results page beyond adding or updating the comparison card
Changing Stripe/payment behavior
Changing upload/extract-text behavior
Adding a new pre-LLM extraction stage just to get dealer name
Constraints
Do not remove or weaken the existing STATE_FEE_REFERENCE and STATE-SPECIFIC FEE RULES prompt sections in server/routes.ts
Use a hard timeout (Promise.race or equivalent) of about 200ms on the warehouse lookup. On timeout, return null and proceed without blocking the LLM call
The warehouse lookup must never block or delay the LLM call beyond the timeout window
If warehouse data is unavailable, the route must behave exactly as it does today
Do not fabricate market stats in the LLM prompt; only inject verified values returned by the helper
Add marketContext as an optional field to the response schema if it is not already present. Do not modify or rename any existing response fields
Use only the existing identifier data already present in the analyze flow. Do not introduce fuzzy matching, normalization pipelines, or new matching logic
Do not expose internal schema names, DB column names, or implementation details in the prompt; use only plain-English phrasing
Ensure the MARKET_INTELLIGENCE block is appended only once per request
Only inject the block when at least one usable statistic is present
All monetary values should be formatted as USD with a dollar sign and no decimals
Handle singular vs plural correctly, e.g. 1 deal vs 847 deals
Only compute comparisons when required values are valid finite numbers
Do not break the free preview or paid full-review flows in client/src/pages/home.tsx
Tasks
Create or update the warehouse market-context helper
Create a helper module at server/marketContext.ts if it does not already exist, or update it if it does. It should:
accept the detected state and whatever exact identifiers are already available in the analyze flow
query warehouse-backed tables/views for:
state total deals
state average doc fee
dealer average score/count only if dealer identity is already safely available
return null when data is unavailable, below threshold, the query fails, or the timeout expires
enforce a hard Promise.race timeout of about 200ms that resolves to null on expiry
use the same dealer sample-size threshold already enforced elsewhere in the codebase; do not invent a new threshold
Pre-fetch market context before the LLM call
In server/routes.ts, inside /api/analyze, call getMarketContext(...) after state detection and before openai.chat.completions.create(...).
Reuse that same marketContext value both:
when building the system prompt
when constructing the final response payload
Do not fetch state market context after the LLM call.
Append a MARKET_INTELLIGENCE block to the existing system prompt
Extend the existing inline prompt string construction inside server/routes.ts — the same place where STATE_FEE_REFERENCE and STATE-SPECIFIC FEE RULES are assembled. Do not introduce a new prompt-building abstraction.
Append the MARKET_INTELLIGENCE block after the existing state fee sections and before the user input content.
When marketContext is non-null and contains at least one usable statistic, append a block like:
MARKET_INTELLIGENCE
In TX, we have analyzed 847 deals
Average doc fee in TX is $312
Use the provided figures exactly as written. Do not estimate, round, or invent additional statistics. Use these figures only when they are materially relevant to the deal under review. Reference them naturally in the summary and reasoning, and do not force their use when they are not helpful.
When marketContext is null, leave the system prompt unchanged.
Important:
If dealer name is not available until post-LLM extracted fields, do not hack around that limitation
In that case, keep dealer-level stats out of the first-pass prompt and treat them as post-LLM enrichment only
Extend the shared response shape only if needed
If the current shared analysis response schema in shared/schema.ts does not already support a marketContext field, add it as an optional field so the server can return the market context object safely.
Keep the shape minimal and explicit. Do not modify or rename any existing response fields.
If useful, include:
stateCode
stateDealCount
stateAvgDocFee
dealerAvgScore
dealerSampleCount
docFeeVsStateAvg
But only include fields actually used by the server/client.
Add or update the “How This Compares” card in client/src/pages/home.tsx
Render it only when result.marketContext exists and has at least one usable statistic.
Use short narrative lines such as:
Doc fee is $340 above the TX average across 847 analyzed deals.
Average doc fee in TX is $312 across 847 analyzed deals.
This dealer's average deal score is 6.2 across 5 analyzed quotes.
Rules:
When both the current deal’s doc fee and state average are available, compute the delta and render above or below accordingly
Only compute and render the delta when both values are valid finite numbers
If only the state average exists, render the average without comparison
Omit any line whose inputs are null, non-finite, or below threshold
Keep singular/plural correct
Do not disrupt the existing free vs paid gating logic
Post-LLM dealer enrichment is allowed if necessary

it is non-fatal
it does not alter the first-pass model prompt retroactively
it is clearly just response/UI enrichment
it does not add fuzzy matching or new normalization logic
