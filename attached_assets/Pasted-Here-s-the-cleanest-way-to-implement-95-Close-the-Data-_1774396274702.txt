Here’s the cleanest way to implement #95 — Close the Data Flywheel without disturbing the current analyze / paywall flow.
Recommended approach
Treat this as three thin additions around the existing pipeline, not a refactor:
Enrich /api/analyze response with optional marketContext
Persist explicit user feedback in a new deal_feedback table
Render two new UI blocks only after results exist:
“How This Compares”
thumbs up / thumbs down feedback
That keeps the current product behavior intact while adding the loop-closing pieces you want.
1) Market context on /api/analyze
Response shape
Add an optional field to the existing analyze response:
type MarketContext = {
  stateAvgDocFee: number | null;
  stateAvgDealScore: number | null;
  stateTotalAnalyses: number;
  dealerAnalysisCount: number | null;
  dealerAvgScore: number | null;
  docFeeVsStateAvg: number | null;
};
And then:
marketContext?: MarketContext;
listingId: number; // or whatever identifier you already return / should return
Threshold rules
Only attach marketContext when:
stateTotalAnalyses >= 10
dealer metrics only included when dealerAnalysisCount >= 3
That means you have two valid states:
State-level only
If Texas has 47 analyses but dealer recognition failed or dealer has only 2 analyses:
return state fields
return dealer fields as null
State + dealer
If thresholds pass for both:
populate everything
That makes the card resilient and prevents UI branching chaos.
Where to query it
Do this after the normal GPT + rules output is assembled, and ideally in parallel with any non-dependent post-processing.
Pattern:
const [analysisResult, marketContext] = await Promise.all([
  runExistingAnalysisPipeline(...),
  getMarketContext(...),
]);
But only if you already know the parsed state, dealerName, and docFee. If those values are only available after analysis normalization, then do:
const analysisResult = await runExistingAnalysisPipeline(...);
const marketContext = await getMarketContext({
  state: analysisResult.state ?? ...,
  dealerName: analysisResult.dealerName ?? ...,
  docFee: analysisResult.docFee ?? ...,
});
That is still fine. The key is:
never block success on warehouse lookup
if warehouse query fails, log it and omit marketContext
So this should be best-effort enrichment, not part of the critical path.
Suggested server helper
Add a dedicated helper, not inline SQL in the route:
async function getMarketContext(params: {
  state?: string | null;
  dealerName?: string | null;
  docFee?: number | null;
}): Promise<MarketContext | null>
Behavior:
no state → return null
query core.state_stats by state
if state threshold < 10 → return null
attempt dealer lookup if dealer is recognized
if dealer threshold < 3 → dealer fields become null
compute docFeeVsStateAvg only when both exist
Good SQL/data mapping rules
From your warehouse objects:
core.state_stats
state average doc fee
average score
total analyses
core.dealers
dealer average score
dealer analysis count
Use whichever canonical identifier already exists for dealers. If matching by raw name, normalize aggressively:
trim
lowercase
collapse whitespace
maybe strip punctuation if your warehouse does too
If dealer recognition is currently fuzzy/fragile, do not invent new matching logic here. Use existing recognized dealer identity if one already comes out of ingestion/analysis.
Important safeguard
Do not let warehouse context break /api/analyze.
Use this pattern:
let marketContext: MarketContext | undefined;

try {
  marketContext = await getMarketContext(... ) ?? undefined;
} catch (err) {
  console.error('[marketContext] failed', err);
}
That matches your “must not slow down or break the existing pipeline” requirement.
2) New dealFeedback table
You wrote “linked to the analysis,” but the proposed column is listingId. I would strongly recommend making sure this points to the actual analysis/submission record users get back from /api/analyze, not a vague “listing.”
If the existing primary record is actually called dealer_submissions, then use that FK explicitly.
Suggested schema
In shared/schema.ts:
export const dealFeedback = pgTable('deal_feedback', {
  id: serial('id').primaryKey(),
  listingId: integer('listing_id')
    .notNull()
    .references(() => dealerSubmissions.id, { onDelete: 'cascade' }),
  rating: boolean('rating').notNull(),
  comment: text('comment'),
  createdAt: timestamp('created_at', { withTimezone: true })
    .notNull()
    .defaultNow(),
});
If your main table is not dealerSubmissions, swap the FK target accordingly.
Validation rules
For POST /api/feedback:
listingId: required integer
rating: required boolean
comment: optional string
trim comment
convert empty string to null
cap comment length, e.g. 280 or 500
reject comment-only submissions without rating
Example Zod:
const feedbackSchema = z.object({
  listingId: z.number().int().positive(),
  rating: z.boolean(),
  comment: z.string().trim().max(500).optional().transform(v => v || undefined),
});
Storage API
In server/storage.ts, add:
createDealFeedback(input: {
  listingId: number;
  rating: boolean;
  comment?: string | null;
}): Promise<{ id: number }>;
Implementation should be a straight insert. No need to overbuild.
Route
POST /api/feedback
Request:
{
  "listingId": 123,
  "rating": true,
  "comment": "Very clear"
}
Response:
{
  "ok": true
}
Recommended anti-spam guard
Allow only one feedback row per listingId per session/user if you have a lightweight way to enforce it. If not, fine to defer. But at minimum the UI should self-disable after successful submission.

Biggest implementation traps
1. Ambiguous FK target
“listingId” is the biggest likely footgun. Make sure it references the exact persisted analysis/submission row created by /api/analyze.
2. Warehouse failures breaking analysis
Do not allow that. Market context must be best-effort.
3. Dealer matching drift
Do not invent new fuzzy matching here. Reuse existing recognized dealer identity if available.
4. UI coupled to full market data
Don’t require every field. Make the card render partial data gracefully.
5. Feedback double-submits
Disable after success.
