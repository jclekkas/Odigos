---

Build the Odigos Auto Data Platform — a unified data ingestion, storage, and analysis system that collects dealer pricing data from two primary pipelines (public marketplace listings and government enforcement records), stores everything in a normalized, ML-ready data warehouse, and exposes APIs for both the consumer-facing product and future data science workloads.

This is our moat. The database architecture matters more than the scrapers. Design it so a data scientist can join any table to any other table, train models on pricing patterns, and we never have to re-architect when we add ML.

## PART 1: DATABASE ARCHITECTURE

Use PostgreSQL. This is NOT a typical web app database — it's a data warehouse that also serves the web app. Design it in three layers:

### Layer 1: Raw Data Lake (schema: `raw`)

Store every piece of ingested data in its original form. Never transform or discard at this layer. This is our audit trail and our reprocessing safety net.

```sql
CREATE SCHEMA raw;

-- Every scraped listing, exactly as we found it
CREATE TABLE raw.marketplace_listings (
  id BIGSERIAL PRIMARY KEY,
  source VARCHAR(50) NOT NULL,            -- 'craigslist', 'cargurus', 'autotrader', 'fb_marketplace'
  source_listing_id VARCHAR(255),          -- The ID on the source platform
  source_url TEXT,
  raw_html TEXT,                           -- Full page HTML for reprocessing
  raw_json JSONB,                          -- Structured extraction attempt
  scraped_at TIMESTAMPTZ DEFAULT NOW(),
  scraper_version VARCHAR(20),             -- Track which version of our scraper produced this
  metro_area VARCHAR(100),
  UNIQUE(source, source_listing_id)
);

-- Every enforcement record, exactly as we found it
CREATE TABLE raw.enforcement_records (
  id BIGSERIAL PRIMARY KEY,
  source VARCHAR(50) NOT NULL,            -- 'ftc', 'cfpb', 'state_ag_ca', 'bbb', 'nhtsa'
  source_record_id VARCHAR(255),
  source_url TEXT,
  raw_html TEXT,
  raw_json JSONB,
  scraped_at TIMESTAMPTZ DEFAULT NOW(),
  scraper_version VARCHAR(20),
  UNIQUE(source, source_record_id)
);

-- Every user-submitted quote analysis (PII-redacted)
CREATE TABLE raw.user_analyses (
  id BIGSERIAL PRIMARY KEY,
  analysis_id UUID DEFAULT gen_random_uuid(),
  submitted_text_redacted TEXT,            -- PII stripped before storage
  state VARCHAR(2),
  vehicle_year INT,
  vehicle_make VARCHAR(50),
  vehicle_model VARCHAR(100),
  analysis_result JSONB,                   -- Full analysis output
  deal_score INT,
  verdict VARCHAR(10),
  flags JSONB,                             -- Array of flag strings
  is_paid BOOLEAN DEFAULT FALSE,
  submitted_at TIMESTAMPTZ DEFAULT NOW(),
  retention_expires_at TIMESTAMPTZ         -- 90-day TTL per privacy policy
);

CREATE INDEX idx_raw_listings_source ON raw.marketplace_listings(source, scraped_at);
CREATE INDEX idx_raw_enforcement_source ON raw.enforcement_records(source, scraped_at);
CREATE INDEX idx_raw_analyses_state ON raw.user_analyses(state, submitted_at);
```

### Layer 2: Normalized / Structured (schema: `core`)

Cleaned, typed, deduplicated data ready for analytics and ML. This is what models train on and APIs query.

```sql
CREATE SCHEMA core;

-- ========== REFERENCE DATA ==========

CREATE TABLE core.states (
  state_code VARCHAR(2) PRIMARY KEY,
  state_name VARCHAR(50) NOT NULL,
  doc_fee_cap DECIMAL(10,2),               -- NULL if uncapped
  doc_fee_cap_type VARCHAR(20),            -- 'hard_cap', 'cpi_indexed', 'safe_harbor', 'uncapped'
  doc_fee_cap_statute TEXT,                -- Legal citation
  doc_fee_cap_effective_date DATE,
  sales_tax_base DECIMAL(5,3),
  sales_tax_max DECIMAL(5,3),              -- Highest local rate in state
  trade_in_credit BOOLEAN,                 -- Does state give sales tax credit on trade-in?
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE core.metro_areas (
  id SERIAL PRIMARY KEY,
  metro_name VARCHAR(100) NOT NULL,
  state_code VARCHAR(2) REFERENCES core.states(state_code),
  population INT,
  avg_local_sales_tax DECIMAL(5,3),
  dealer_count_estimate INT
);

-- ========== DEALER ENTITIES ==========

-- Deduplicated dealer directory built from all sources
CREATE TABLE core.dealers (
  id BIGSERIAL PRIMARY KEY,
  dealer_name VARCHAR(255) NOT NULL,
  dealer_name_normalized VARCHAR(255),     -- Lowercase, stripped, for fuzzy matching
  city VARCHAR(100),
  state_code VARCHAR(2) REFERENCES core.states(state_code),
  zip_code VARCHAR(10),
  metro_area_id INT REFERENCES core.metro_areas(id),
  first_seen_at TIMESTAMPTZ,
  last_seen_at TIMESTAMPTZ,
  listing_count INT DEFAULT 0,
  enforcement_count INT DEFAULT 0,
  complaint_count INT DEFAULT 0,
  avg_deal_score DECIMAL(5,2),             -- Computed from their listings
  risk_score DECIMAL(5,2),                 -- ML-computed later
  UNIQUE(dealer_name_normalized, city, state_code)
);

CREATE INDEX idx_dealers_state ON core.dealers(state_code);
CREATE INDEX idx_dealers_metro ON core.dealers(metro_area_id);
CREATE INDEX idx_dealers_name ON core.dealers(dealer_name_normalized);

-- ========== LISTINGS (Structured from raw) ==========

CREATE TABLE core.listings (
  id BIGSERIAL PRIMARY KEY,
  raw_listing_id BIGINT REFERENCES raw.marketplace_listings(id),
  dealer_id BIGINT REFERENCES core.dealers(id),
  source VARCHAR(50),

  -- Vehicle
  vehicle_year INT,
  vehicle_make VARCHAR(50),
  vehicle_model VARCHAR(100),
  vehicle_trim VARCHAR(100),
  vehicle_type VARCHAR(20),                -- 'new', 'used', 'cpo'
  mileage INT,
  vin VARCHAR(17),

  -- Pricing (the money columns — this is what ML cares about)
  listed_price DECIMAL(10,2),
  price_type VARCHAR(20),                  -- 'msrp', 'sale_price', 'otd', 'call_for_price', 'monthly_payment'
  has_otd_price BOOLEAN,
  otd_price DECIMAL(10,2),                 -- If available
  monthly_payment DECIMAL(10,2),
  apr_disclosed BOOLEAN,
  apr_value DECIMAL(5,3),
  loan_term_months INT,
  down_payment DECIMAL(10,2),

  -- Fees (each as a separate decimal — ML features)
  doc_fee DECIMAL(10,2),
  doc_fee_over_state_cap BOOLEAN,
  doc_fee_overage DECIMAL(10,2),           -- How much over cap (NULL if not over)
  market_adjustment DECIMAL(10,2),
  reconditioning_fee DECIMAL(10,2),
  prep_fee DECIMAL(10,2),
  electronic_filing_fee DECIMAL(10,2),
  total_disclosed_fees DECIMAL(10,2),      -- Sum of all non-tax fees

  -- Add-ons
  has_addons BOOLEAN,
  addon_names TEXT[],                       -- Array: ['paint_protection', 'nitrogen', 'vin_etch']
  addon_total DECIMAL(10,2),
  addons_presented_as_mandatory BOOLEAN,

  -- Analysis output
  deal_score INT,
  verdict VARCHAR(10),
  flags TEXT[],                             -- Array of flag strings
  flag_count INT,

  -- Computed features for ML
  price_per_mile DECIMAL(10,4),            -- listed_price / mileage (used cars)
  fee_to_price_ratio DECIMAL(5,4),         -- total_disclosed_fees / listed_price
  days_on_market INT,                      -- If available from source

  -- Metadata
  listing_date DATE,
  scraped_at TIMESTAMPTZ,
  analyzed_at TIMESTAMPTZ,

  UNIQUE(raw_listing_id)
);

CREATE INDEX idx_listings_dealer ON core.listings(dealer_id);
CREATE INDEX idx_listings_state ON core.listings(dealer_id);  -- Join through dealers
CREATE INDEX idx_listings_vehicle ON core.listings(vehicle_make, vehicle_model, vehicle_year);
CREATE INDEX idx_listings_score ON core.listings(deal_score);
CREATE INDEX idx_listings_date ON core.listings(listing_date);
CREATE INDEX idx_listings_fees ON core.listings(doc_fee, market_adjustment, reconditioning_fee);

-- ========== ENFORCEMENT & COMPLAINTS ==========

CREATE TABLE core.enforcement_actions (
  id BIGSERIAL PRIMARY KEY,
  raw_record_id BIGINT REFERENCES raw.enforcement_records(id),
  dealer_id BIGINT REFERENCES core.dealers(id),   -- Linked to dealer entity when possible
  source VARCHAR(50),
  dealer_name_raw VARCHAR(255),                     -- Original name from source
  state_code VARCHAR(2) REFERENCES core.states(state_code),
  city VARCHAR(100),

  action_date DATE,
  violation_type VARCHAR(50),                       -- Standardized taxonomy (see below)
  violation_subtype VARCHAR(100),
  violation_description TEXT,
  fine_amount DECIMAL(12,2),
  settlement_amount DECIMAL(12,2),
  is_resolved BOOLEAN,
  source_url TEXT,

  analyzed_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE core.consumer_complaints (
  id BIGSERIAL PRIMARY KEY,
  raw_record_id BIGINT REFERENCES raw.enforcement_records(id),
  source VARCHAR(50),                               -- 'cfpb', 'bbb', 'nhtsa', 'state_ag'
  complaint_date DATE,
  state_code VARCHAR(2) REFERENCES core.states(state_code),
  company_name VARCHAR(255),
  dealer_id BIGINT REFERENCES core.dealers(id),

  complaint_type VARCHAR(50),                       -- Standardized taxonomy
  complaint_subtype VARCHAR(100),
  company_response VARCHAR(100),
  consumer_narrative_available BOOLEAN,

  analyzed_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_enforcement_state ON core.enforcement_actions(state_code, action_date);
CREATE INDEX idx_enforcement_dealer ON core.enforcement_actions(dealer_id);
CREATE INDEX idx_enforcement_type ON core.enforcement_actions(violation_type);
CREATE INDEX idx_complaints_state ON core.consumer_complaints(state_code, complaint_date);
CREATE INDEX idx_complaints_dealer ON core.consumer_complaints(dealer_id);

-- ========== AGGREGATED STATS (Materialized views for fast API responses) ==========

CREATE MATERIALIZED VIEW core.state_stats AS
SELECT
  s.state_code,
  s.state_name,
  s.doc_fee_cap,
  s.doc_fee_cap_type,
  COUNT(DISTINCT l.id) AS total_listings_scanned,
  AVG(l.deal_score) AS avg_deal_score,
  AVG(l.doc_fee) AS avg_doc_fee,
  AVG(l.total_disclosed_fees) AS avg_total_fees,
  SUM(CASE WHEN l.has_otd_price = FALSE THEN 1 ELSE 0 END)::DECIMAL / NULLIF(COUNT(l.id), 0) AS pct_missing_otd,
  SUM(CASE WHEN l.doc_fee_over_state_cap = TRUE THEN 1 ELSE 0 END)::DECIMAL / NULLIF(COUNT(l.id), 0) AS pct_doc_fee_over_cap,
  COUNT(DISTINCT e.id) AS enforcement_action_count,
  SUM(e.fine_amount) AS total_fines,
  COUNT(DISTINCT c.id) AS complaint_count
FROM core.states s
LEFT JOIN core.dealers d ON d.state_code = s.state_code
LEFT JOIN core.listings l ON l.dealer_id = d.id
LEFT JOIN core.enforcement_actions e ON e.state_code = s.state_code
LEFT JOIN core.consumer_complaints c ON c.state_code = s.state_code
GROUP BY s.state_code, s.state_name, s.doc_fee_cap, s.doc_fee_cap_type;

CREATE MATERIALIZED VIEW core.national_stats AS
SELECT
  COUNT(DISTINCT l.id) AS total_listings_scanned,
  COUNT(DISTINCT ua.id) AS total_user_analyses,
  AVG(l.deal_score) AS avg_deal_score,
  AVG(l.total_disclosed_fees) AS avg_hidden_fees,
  COUNT(DISTINCT e.id) AS total_enforcement_actions,
  SUM(e.fine_amount) AS total_documented_fines
FROM core.listings l
CROSS JOIN (SELECT id FROM raw.user_analyses) ua
LEFT JOIN core.enforcement_actions e ON TRUE;
```

### Layer 3: ML Feature Store (schema: `ml`)

Pre-computed features for model training. A data scientist should be able to `SELECT * FROM ml.listing_features` and start training immediately.

```sql
CREATE SCHEMA ml;

-- One row per listing, all features as floats for direct model ingestion
CREATE TABLE ml.listing_features (
  listing_id BIGINT PRIMARY KEY REFERENCES core.listings(id),

  -- Vehicle features
  vehicle_age INT,                          -- current_year - vehicle_year
  is_new BOOLEAN,
  is_cpo BOOLEAN,
  make_popularity_rank INT,                 -- Computed from listing volume

  -- Price features
  listed_price DECIMAL(10,2),
  price_vs_state_avg DECIMAL(10,4),         -- Ratio to state average for same make/model/year
  price_vs_national_avg DECIMAL(10,4),
  has_otd_price BOOLEAN,
  price_type_is_monthly BOOLEAN,

  -- Fee features (all numeric, NULL = not detected)
  doc_fee DECIMAL(10,2),
  doc_fee_vs_state_cap DECIMAL(10,4),       -- Ratio (>1.0 means over cap)
  market_adjustment DECIMAL(10,2),
  reconditioning_fee DECIMAL(10,2),
  prep_fee DECIMAL(10,2),
  total_fees DECIMAL(10,2),
  fee_to_price_ratio DECIMAL(5,4),
  addon_count INT,
  addons_mandatory BOOLEAN,

  -- Transparency features
  apr_disclosed BOOLEAN,
  loan_term_disclosed BOOLEAN,
  down_payment_disclosed BOOLEAN,
  transparency_score INT,                   -- 0-100, how many pricing elements are disclosed

  -- Dealer features (joined from core.dealers)
  dealer_listing_count INT,
  dealer_avg_score DECIMAL(5,2),
  dealer_enforcement_count INT,
  dealer_complaint_count INT,
  dealer_risk_score DECIMAL(5,2),

  -- Geographic features
  state_code VARCHAR(2),
  state_doc_fee_capped BOOLEAN,
  state_sales_tax_rate DECIMAL(5,3),
  metro_population INT,

  -- Target variable (for supervised learning)
  deal_score INT,
  verdict_numeric INT,                      -- GREEN=2, YELLOW=1, RED=0

  -- Metadata
  computed_at TIMESTAMPTZ DEFAULT NOW()
);

-- Time-series features for trend modeling
CREATE TABLE ml.weekly_market_stats (
  id SERIAL PRIMARY KEY,
  week_start DATE,
  state_code VARCHAR(2),
  vehicle_make VARCHAR(50),
  avg_listed_price DECIMAL(10,2),
  avg_doc_fee DECIMAL(10,2),
  avg_deal_score DECIMAL(5,2),
  avg_fee_to_price_ratio DECIMAL(5,4),
  listing_count INT,
  pct_missing_otd DECIMAL(5,4),
  pct_over_cap DECIMAL(5,4),
  UNIQUE(week_start, state_code, vehicle_make)
);

-- Dealer risk model training data
CREATE TABLE ml.dealer_features (
  dealer_id BIGINT PRIMARY KEY REFERENCES core.dealers(id),
  total_listings INT,
  avg_deal_score DECIMAL(5,2),
  avg_fee_to_price_ratio DECIMAL(5,4),
  pct_listings_missing_otd DECIMAL(5,4),
  pct_listings_over_cap DECIMAL(5,4),
  pct_listings_with_mandatory_addons DECIMAL(5,4),
  enforcement_count INT,
  complaint_count INT,
  total_fines DECIMAL(12,2),
  years_active INT,
  state_code VARCHAR(2),
  metro_population INT,
  risk_score DECIMAL(5,2),                  -- Target for risk model
  computed_at TIMESTAMPTZ DEFAULT NOW()
);
```

## PART 2: INGESTION PIPELINES

### Pipeline A: Marketplace Listing Scraper

Build a scraper service at `/services/scraper/` that:

1. Scrapes publicly visible dealer listings from Craigslist (dealer section), CarGurus, and Facebook Marketplace (public dealer posts).
2. Stores raw HTML + extracted JSON in `raw.marketplace_listings`.
3. Runs the Odigos analysis logic on extracted data.
4. Writes normalized results to `core.listings` and `core.dealers` (upsert dealers by normalized name + city + state).
5. Computes ML features and writes to `ml.listing_features`.

**Scraping rules:**
- Respect robots.txt for each source.
- 2-5 second delay between requests.
- Rotating user agents.
- Start with Craigslist Los Angeles, then expand to 10 metros, then 50.
- Target: 500+ listings/day within first month.

**Dealer entity resolution:**
- Normalize dealer names: lowercase, strip "inc", "llc", "auto group", common suffixes.
- Match on (normalized_name, city, state). If close match exists (Levenshtein distance < 3), link to existing dealer rather than creating duplicate.

### Pipeline B: Enforcement & Complaint Ingestion

Build an ingestion service at `/services/enforcement/` that:

1. **CFPB Complaints:** Weekly API pull of new "Vehicle loan or lease" complaints. Store raw in `raw.enforcement_records`, normalize into `core.consumer_complaints`. Try to match company name to existing `core.dealers` entries.

2. **FTC Enforcement:** Weekly scrape of `ftc.gov/legal-library/browse/cases-proceedings` filtered to motor vehicle dealers. Extract case details into `core.enforcement_actions`.

3. **State AG Press Releases:** Weekly scrape of top 10 state AG news pages. Parse press releases mentioning auto dealers. Extract into `core.enforcement_actions` with NLP classification of violation type.

4. **BBB:** Monthly scrape of BBB auto dealer profiles. Update `core.dealers` with complaint counts and ratings.

**Violation type taxonomy** (standardize all sources to these):
- `deceptive_pricing`, `hidden_fees`, `doc_fee_overcharge`, `mandatory_addons`, `yo_yo_financing`, `bait_and_switch`, `payment_packing`, `interest_rate_markup`, `title_washing`, `other`

### Pipeline C: User Analysis Storage

When a user runs an analysis on odigosauto.com, the existing analysis flow should ALSO:
1. PII-redact the submitted text (strip names, emails, phone numbers, addresses).
2. Write the redacted text + analysis result to `raw.user_analyses`.
3. Extract structured fields (vehicle, state, fees detected) into `core.listings` with `source = 'user_submitted'`.
4. This means user analyses contribute to the same dataset as scraped listings — growing the moat with every use.

## PART 3: API LAYER

Build Express API routes at `/services/api/`:

### Public API (powers odigosauto.com)
- `GET /api/stats` — National stats from `core.national_stats` materialized view. Powers homepage counter.
- `GET /api/stats/:state` — State stats from `core.state_stats`. Powers state pages.
- `GET /api/enforcement/:state` — Recent enforcement actions for a state.
- `GET /api/trends/:state` — Weekly market trends from `ml.weekly_market_stats`.
- `GET /api/dealer/:dealerNameNormalized` — Dealer profile: listing count, avg score, enforcement history.

### Internal API (for future ML pipelines and admin)
- `POST /api/internal/refresh-materialized-views` — Recompute all materialized views.
- `POST /api/internal/compute-ml-features` — Recompute `ml.listing_features` and `ml.dealer_features`.
- `GET /api/internal/export/:table?format=csv` — CSV export of any `ml.*` table for notebook-based model training.
- `GET /api/internal/schema` — Returns current row counts for all tables across all schemas.

## PART 5: FUTURE ML USE CASES (Don't build these now — just make sure the schema supports them)

The schema above is designed so a data scientist can later build:

1. **Fair Price Model** — Given vehicle (make, model, year, mileage) + state + metro, predict what a fair OTD price should be. Training data: `ml.listing_features` joined with `core.listings`.

2. **Dealer Risk Score** — Given dealer history (avg score, enforcement count, complaint count, fee patterns), compute a risk score 0-100. Training data: `ml.dealer_features`.

3. **Fee Anomaly Detection** — Flag listings where fees are statistical outliers for their state/metro/vehicle type. Uses `ml.weekly_market_stats` for baseline distributions.

4. **Quote Quality Predictor** — Given a user-submitted quote, predict the likelihood of hidden fees before full analysis. Training data: `raw.user_analyses` + `ml.listing_features`.

5. **Market Trend Forecasting** — Predict fee trends by state/metro. Training data: `ml.weekly_market_stats` time series.

Don't build any models now. Just make sure every table has the right columns, indexes, and foreign keys so that when we're ready, a data scientist can `SELECT * FROM ml.listing_features LIMIT 100000` and start a Jupyter notebook immediately.
