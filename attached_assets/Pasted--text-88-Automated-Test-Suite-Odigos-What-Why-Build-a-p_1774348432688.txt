88 - Automated Test Suite (Odigos)

What & Why
Build a production-grade, layered automated test suite that gives real engineering confidence across Odigos. This is a trust-sensitive consumer protection product where regressions in analysis flow, upload handling, API contracts, or free/paid boundaries are unacceptable. The suite must be structured around business impact, not test volume, and must remain fast enough to be practical for everyday use and future CI.

Done looks like
- npm run test runs all unit + component + API integration tests and passes
- npm run test:e2e runs Playwright E2E tests and passes
- All Tier 1 flows (analyze flow, rule engine, upload, API contracts, free/paid boundary) are covered with real assertions
- Tier 2 surfaces (landing page, representative article page, navigation, CTAs) have smoke-level rendering and interaction coverage
- A clear README/replit.md/TESTING.md section explains how to run each layer
- No real OpenAI, Stripe, or other external API calls are made in any test
- Unit + integration suite completes in under 60 seconds total
- E2E is separate, minimal, stable, and focused on highest-signal user journeys

Out of scope
- 100% code coverage as a goal
- Exhaustive testing of decorative or styling-only UI
- Deep testing of admin/dashboard surfaces
- CI/CD integration (GitHub Actions etc.) in this task
- Replit-specific integrations unrelated to core product behavior

Test Quality Rules (CRITICAL)
- Test behavior, not implementation details
- Do NOT test private state or internal function calls
- Do NOT rely on fragile snapshot-heavy testing
- Prefer explicit assertions over vague object matching
- Keep tests deterministic: no randomness, no real network calls
- Each test should fail for a clear, meaningful reason
- Avoid over-mocking the app into unrealistic behavior
- Keep runtime fast and practical

Mocking Rules (CRITICAL)
- All external services MUST be mocked:
  - OpenAI chat/completions or SDK analysis calls
  - OpenAI vision/image extraction calls
  - Stripe, if touched
  - pdf-parse
- Mocks must return realistic structured responses that match production shape
- Do NOT mock internal business logic that should be tested for real, such as:
  - rule engine
  - fee lookup logic
  - validation helpers
  - deterministic utility functions
- Prefer reusable module-level mocks/shared helpers over one-off inline hacks

Prioritization Rule (CRITICAL)
Prioritize test coverage by business impact:

Tier 1 — Must never break
- analyze flow
- rule engine / verdict logic
- upload and extraction handling
- API contracts
- free/paid boundary
- primary conversion CTAs
- critical error handling

Tier 2 — Should not break
- landing page
- representative article/SEO page
- about/privacy or similar trust pages
- navigation and routing
- in-article CTA flows
- FAQ/accordion behavior
- accessibility-critical semantics

Tier 3 — Low priority
- decorative UI
- styling-only elements
- low-impact buttons with no business significance

Focus heavily on Tier 1 and Tier 2. Do NOT spend time exhaustively testing low-value UI.

Tasks

1. Install and configure the test stack
Set up:
- Vitest
- React Testing Library
- @testing-library/user-event
- @testing-library/jest-dom
- jsdom
- supertest or the lightest credible HTTP testing tool
- Playwright for E2E

Create vitest.config.ts with separate projects/environments for:
- Node/server tests
- jsdom/component tests

Create playwright.config.ts appropriate for the repo’s local dev server.

Add the repo-appropriate run commands/scripts for:
- test
- test:unit
- test:components
- test:e2e

Do not edit package.json directly unless that is actually how this repo expects scripts to be managed. Use the repo’s actual conventions.

2. Unit tests: rule engine
Write thorough unit tests for server/ruleEngine.ts covering real behavior and edge cases, including:
- correct verdict (GREEN / YELLOW / RED) from applyRuleEngine
- hard override behavior when doc fee exceeds state cap via checkDocFeeCap
- add-on threshold boundaries at $300 and $500
- market adjustment / ADM triggering mandatory RED
- vague fee keyword detection
- confidence calculation when OTD + APR + term are all present versus missing

These tests must run the real rule engine logic, not a mock.

3. Unit tests: state fee lookup and state detection
Write unit tests for server/stateFeeLookup.ts covering:
- ZIP code → state detection
- geographic keyword detection
- unknown or missing input returning null
- getStateFeeData returning correct caps/ranges from state_fee_reference.json for a representative sample of states

Use explicit assertions against the expected values.

4. Unit tests: supporting helpers
Write unit tests for:
- server/piiRedact.ts
  - PII patterns are redacted
  - clean text passes through
- shared/schema.ts
  - Zod validation shapes pass/fail correctly for detectedFieldsSchema
  - relevant insert schemas validate and reject bad inputs correctly

Only test pure/deterministic helpers that materially affect product behavior.

5. API integration tests: POST /api/analyze
Using Vitest in Node environment and supertest (or equivalent), write integration tests for POST /api/analyze.

Mock OpenAI so no real API calls occur.

Cover:
- happy path returns correct success shape
- missing dealerText body returns 400
- malformed JSON returns 400
- rule engine override fires when mocked LLM output includes a doc fee cap violation
- response includes state-specific fields when a state is detected
- LLM failure or malformed LLM response is handled gracefully and returns a safe error path

API Response Assertions (IMPORTANT)
Explicitly assert:
- required fields exist and have correct types
- critical outputs are not unexpectedly null/undefined
- enum-like values are correct and constrained (GREEN/YELLOW/RED etc.)
- success response structure is consistent

6. API integration tests: POST /api/extract-text and GET /api/state-fee/:state
Test POST /api/extract-text with:
- unsupported MIME type returns 400
- oversized file returns 400
- valid PDF buffer path using mocked pdf-parse
- valid image buffer path using mocked OpenAI vision
- PDF with insufficient extracted text returns the expected error
- extraction failure is handled gracefully without crashing

Test GET /api/state-fee/:state with:
- known state returns correct fee object
- unknown state returns 404

7. Component/page tests: analyzer page
Using React Testing Library, test client/src/pages/home.tsx for high-value user behavior:

Cover:
- page renders without crash
- textarea for dealer text is present
- Analyze button is present and has an accessible name
- Analyze button is disabled when input is empty
- Analyze button becomes disabled during loading to prevent duplicate submission
- file upload input has an accessible label
- uploading a file triggers extract-text flow (mock fetch)
- form submission calls /api/analyze (mock fetch)
- loading state appears while request is pending
- results section appears when mock returns success
- error state appears when API fails
- user-safe error handling appears if backend/LLM path fails

Test behavior from the user perspective, not component internals.

8. Component/page tests: Tier 2 pages
Add smoke-level rendering and interaction coverage for:
- landing page
- one representative article/SEO page
- about page and/or privacy page
- representative navigation path

Verify:
- no crash
- critical headings/sections exist
- primary CTA exists and is descriptive
- CTA points to or routes to /analyze where appropriate
- main landmark exists
- critical trust/conversion content is present

Do not over-test static copy word-for-word unless necessary.

9. Accessibility-critical checks
Within component tests, add targeted accessibility assertions for important surfaces:
- main landmark exists
- primary CTA buttons have descriptive accessible text
- upload input has a label
- form controls are labeled
- Analyze button has accessible name
- accordions/FAQ elements expose aria-expanded and aria-controls where applicable
- key interactions are keyboard-operable

Do not attempt full accessibility certification. Focus on meaningful automated checks for critical UX.

10. E2E setup and interception
Install and configure Playwright.

Set up route interception so no real external services are called during E2E:
- OpenAI analysis routes mocked
- OpenAI vision/extraction routes mocked
- Stripe/checkout routes mocked if touched

Keep the setup stable and reusable.

11. E2E tests: highest-signal user journeys only
Write a minimal, stable set of Playwright tests. Do NOT duplicate lower-level coverage unnecessarily.

Target 4–6 high-signal flows max, covering:
- user lands on homepage → navigates to /analyze → pastes dealer quote → clicks Analyze → sees verdict/result
- user submits empty form → sees validation error
- user uploads supported file → extracted text populates textarea → analysis proceeds
- user uploads unsupported file type → sees error
- clicking the primary upgrade/paid CTA from a result state navigates to or triggers checkout flow safely
- navigating from a representative article page to the analyzer via CTA lands correctly on /analyze

E2E Scope Rule
- E2E should validate only the most important real user journeys
- Do NOT bloat E2E with cases already well-covered by unit/integration/component tests
- Optimize for stability, speed, and signal

12. Documentation
Add a concise testing section to replit.md and optionally TESTING.md covering:
- testing stack used
- how to run each layer:
  - test
  - test:unit
  - test:components
  - test:e2e
- what each layer covers
- mocking approach
- known gaps or future test opportunities

Performance Constraints
- Unit + integration + component test run must complete in under 60 seconds total
- Avoid slow waits/timeouts unless truly necessary
- Keep E2E focused and separate
- Prefer fast deterministic mocks over heavy setup

Relevant files
- server/ruleEngine.ts
- server/stateFeeLookup.ts
- server/extractText.ts
- server/routes.ts
- server/piiRedact.ts
- server/state_fee_reference.json
- shared/schema.ts
- client/src/pages/home.tsx
- client/src/App.tsx
- client/src/pages/landing.tsx
- vite.config.ts

Deliverables
When finished, report:
1. exact test stack added
2. all files created/modified
3. commands/scripts added for each layer
4. coverage summary by layer:
   - unit
   - API/integration
   - component/page
   - accessibility
   - E2E
5. what Tier 1 and Tier 2 behavior is now covered
6. any meaningful gaps that remain
7. any bugs discovered during implementation
