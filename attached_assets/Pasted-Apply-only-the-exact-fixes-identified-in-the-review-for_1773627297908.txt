1) CRITICAL FIX — remove broken PDF vision fallback
Problem:
In server/extractText.ts, when pdf-parse returns sparse text, the code currently falls back by sending the raw PDF buffer to the vision helper as if it were image/png. That is invalid and will not work reliably.

Required change:
- Do NOT attempt vision fallback for PDFs
- If pdf-parse returns insufficient extracted text for a PDF, throw a descriptive error instead
- Let the route return a calm 422 message to the client

Implementation intent:
Replace the current PDF fallback behavior with something equivalent to:
- if trimmed extracted PDF text is under the threshold, throw an error like:
  "PDF contained insufficient extractable text"

Important:
- Keep vision extraction only for actual image files
- Do not invent a new PDF rendering pipeline
- Do not add new dependencies

2) MEDIUM-RISK FIX — catch Multer file-size errors with calm copy
Problem:
If a user bypasses client-side validation and uploads a file over 10 MB, Multer throws before the async route handler runs. That means the current calm message is bypassed and Express returns a technical error string.

Required change:
Wrap the multer middleware invocation so Multer errors are intercepted and converted into calm JSON responses before the main handler runs.

Desired behavior:
- LIMIT_FILE_SIZE returns a calm message like:
  "That file is too large to process. Please use a file under 10 MB."
- Other multer errors return a generic calm message like:
  "Something went wrong processing the file."

Important:
- Keep the same endpoint: POST /api/extract-text
- Keep memoryStorage()
- Keep no-disk-write behavior
- Keep server-side MIME validation in place
