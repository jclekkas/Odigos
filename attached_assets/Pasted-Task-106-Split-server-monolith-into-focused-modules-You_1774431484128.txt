Task #106 — Split server monolith into focused modules

You are doing a pure refactor. Do not change behavior, endpoint paths, response shapes, business logic, frontend code, shared schema, or test files unless absolutely required for import-path compatibility. The goal is to make the backend navigable while preserving identical runtime behavior.

Primary objective
Split:
- server/metrics.ts (currently monolithic)
- server/routes.ts (currently monolithic)

into focused modules with thin assembler / re-export shims so the rest of the app keeps working unchanged.

Hard constraints
- NO API path changes
- NO response shape changes
- NO business logic rewrites
- NO frontend changes
- NO shared schema changes
- Existing tests must pass unchanged
- TypeScript must compile cleanly
- App must start and behave identically
- Preserve existing import compatibility wherever possible

Required output structure

1) Split metrics into modules
Create:
- server/metrics/events.ts
- server/metrics/technical.ts
- server/metrics/bi.ts
- server/metrics/experiments.ts
- server/metrics/index.ts

Module responsibilities:
- events.ts
  - KV storage helpers

  - getImportedSessionIds

- technical.ts
  - getMetricsSummary
  - getTechnicalSummary
  - getPiiExpiryStatus
  - getPaymentCountLastNHours
  - all associated interfaces/types used by these functions

- bi.ts
  - all getBI* functions:
    - getBIFunnel
    - getBIPageAttribution
    - getBIUserBehavior
    - getBIDealOutcome
    - getBIGeographic
    - getBIAcquisition
    - getBIRevenue
    - getBIFallout
  - shared cache helpers
  - shared BI filter helpers
  - any BI-only types

- experiments.ts
  - getExperimentStats
  - experiment-related types

- index.ts
  - re-export every public symbol required by existing imports

Compatibility requirement:
- Keep server/metrics.ts as a shim that simply re-exports from ./metrics/index if removing it would break existing imports.
- Existing imports like import ... from "./metrics" or "../metrics" must continue to work without needing broad call-site edits.

2) Split routes into domain routers
Create:
- server/routes/analyze.ts
- server/routes/payments.ts
- server/routes/tracking.ts
- server/routes/admin.ts
- server/routes/bi.ts
- server/routes/reference.ts
- server/routes/index.ts

Routing split:
- analyze.ts
  - /api/analyze
  - /api/feedback
  - /api/extract-text

- payments.ts
  - /api/stripe-status
  - /api/checkout
  - /api/create-checkout-session

- tracking.ts
  - /api/track

- admin.ts

- index.ts
  - assembles all sub-routers
  - exports a single registerRoutes(app) function
  - must match the existing signature expected by server/index.ts

Compatibility requirement:
- Keep server/routes.ts as a shim that re-exports registerRoutes from ./routes/index if removing it would break existing imports.

Implementation guidance
- Reuse existing logic exactly; move code, don’t redesign it.
- Extract shared helpers only when necessary to avoid duplication caused by the split.
- Keep middleware ordering, env checks, Stripe/raw-body handling, admin auth behavior, and route registration semantics identical.
- Be especially careful with:
  - webhook parsing behavior
  - any route-specific middleware
  - imported singleton state/caches
  - cross-module type dependencies
  - metrics cache behavior
  - BI filter normalization
  - health / sitemap / robots behavior

- Prefer minimal necessary import updates over broad cleanup.
- Do not opportunistically rename unrelated symbols or reformat the entire codebase.
- Preserve comments where useful, especially around subtle behavior.
