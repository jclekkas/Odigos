#107 - Fix Analysis Completion UX

Implement this task EXACTLY as specified. Keep changes minimal, surgical, and consistent with existing patterns. Do NOT introduce unrelated refactors.

## What & Why
Two UX failures undermine trust in the analysis flow:

1. Misleading time copy
The form tells users results arrive "in seconds" and a CTA component says "about 10 seconds," but real latency is 40+ seconds. First-time users waiting silently conclude something broke.

2. Silent completion with no scroll
When analysis finishes, the button quietly reverts to "Analyze Deal" while results appear below the fold. Users who haven't scrolled down assume the action failed and leave.

## Done looks like
- Time copy accurately reflects real latency in both the form subtext and the article CTA component
- Use honest language such as "usually under a minute"
- On analysis completion, the button enters a brief completion state like "Done — see your results ↓" for about 1–2 seconds before reverting to idle
- The page auto-scrolls to the results section immediately after analysis completes successfully
- Results are visible without any manual scrolling
- Loading state text and button behavior are otherwise unchanged

## Out of scope
- Backend speed improvements
- Live progress indicators
- Streaming responses
- Any changes to the results UI itself
- Unrelated copy cleanup or refactors

## Tasks

Update the misleading analysis timing copy in both places below to accurate, honest language that sets a realistic expectation.

- when the request finishes successfully, switch the button into a brief completion state
- completion state text should be:
  - `Done — see your results ↓`
  - or a very close equivalent

- after that, revert the button to its normal idle label

Important:
- this completion state should happen only on successful completion
- do not change error behavior
- do not change disabled/loading behavior during submission
- loading state text and behavior should otherwise remain unchanged

### 3. Auto-scroll to the results section
Immediately after a successful analysis completes:
- programmatically scroll the results section into view
- use smooth scrolling

- do not scroll on error
- do not require any user interaction

Implementation guidance:
- add a ref to the results container
- detect the transition from loading -> successful result available
- trigger `scrollIntoView({ behavior: "smooth", block: "start" })`
- coordinate the brief completion button state with the scroll so the user gets both:
  - a visible completion signal
  - immediate arrival at the results
