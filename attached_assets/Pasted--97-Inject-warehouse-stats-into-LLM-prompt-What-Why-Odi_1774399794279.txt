#97 - Inject warehouse stats into LLM prompt
What & Why
Odigos already injects state fee reference data into the GPT-4o system prompt inside server/routes.ts before the OpenAI call. However, it does not currently inject warehouse-derived market context such as state-level average doc fee volume or dealer historical averages. That means the AI summary and reasoning can say a fee “seems high” without grounding that claim in Odigos platform data.
This task extends the existing prompt-injection flow so the model can reference warehouse stats when available. The goal is to ground natural-language analysis in actual platform data without changing behavior when warehouse data is unavailable.
Done looks like
/api/analyze fetches warehouse market context before the GPT-4o call, after state detection and before openai.chat.completions.create(...)
The system prompt includes a MARKET_INTELLIGENCE block only when market context exists
The block gives the model plain-English figures it can cite directly, such as:
state code
number of analyzed deals in that state
state average doc fee, when present
dealer historical average score, when the existing threshold is met
If the warehouse lookup fails, times out, or returns insufficient data, the market block is omitted and the LLM path behaves exactly as before
The API response includes the same market context object used for prompt injection, so the frontend can render it
client/src/pages/home.tsx renders a new “How This Compares” card only when market context is present
That card uses short narrative phrases, not label/value rows
The existing state-fee injection, free preview flow, paid unlock flow, and response schema validation continue to work
Out of scope
Changing the existing state fee prompt injection logic except to append warehouse-based context
Fuzzy dealer matching
Fine-tuning or retraining the model
Any redesign of the results page beyond adding the comparison card
Changing Stripe/payment behavior
Changing upload/extract-text behavior
Constraints
Do not remove or weaken the existing STATE_FEE_REFERENCE and STATE-SPECIFIC FEE RULES prompt sections in server/routes.ts
Keep the warehouse lookup on a hard timeout guard of about 200ms so analysis latency does not regress materially
If warehouse data is unavailable, the route must behave exactly as it does today
Do not fabricate market stats in the LLM prompt; only inject verified values returned by the helper
Keep the result JSON valid against the shared analysis response schema
Do not break the free preview or paid full-review flows in client/src/pages/home.tsx
Tasks
1) Create a warehouse market-context helper
Add a new server helper module for market-context lookup. Since server/marketContext.ts does not currently exist in this repo, create it rather than assuming it is already there. The helper should:
accept the detected state and whatever exact dealer identifiers are already available in the analyze flow
query warehouse-backed tables/views for:
state code
total analyzed deals in that state
average doc fee in that state, when available
dealer average score and dealer sample count, when threshold is met
return null when data is unavailable, below threshold, or the query fails
enforce a timeout guard of about 200ms
2) Pre-fetch market context before the LLM call
In server/routes.ts, inside /api/analyze, fetch market context after state detection and before the OpenAI API call. Reuse that same marketContext value both:
when building the system prompt
when constructing the final response payload
Do not fetch it after the LLM call.
3) Append a MARKET_INTELLIGENCE block to the existing system prompt
Do not replace the current prompt-building approach. Extend it.
When marketContext is non-null, append a block like:
MARKET_INTELLIGENCE
In TX, we have analyzed 847 deals
Average doc fee in TX is $312
This dealer’s average deal score is 6.2 across 5 analyzed quotes
Instruct the model to reference these figures in summary and reasoning when materially relevant, and not to force their use when they do not meaningfully help evaluate the deal.
When marketContext is null, leave the system prompt unchanged.
4) Extend the shared response shape only if needed
If the current shared analysis response schema does not already support a marketContext field, add it in the shared schema so the server can return the market context object to the client safely.
Keep the shape minimal and explicit. Do not add unused fields.
5) Add a new results card on client/src/pages/home.tsx
There is currently no MarketContextCard or “How This Compares” section in home.tsx, so create one.
Render it only when result.marketContext exists.
Use short narrative lines such as:
“Doc fee is $340 above the TX average across 847 analyzed deals.”
“This dealer’s average deal score is 6.2 across 5 analyzed quotes.”
Omit any line whose inputs are null or below threshold.
Place the card in the results area near the summary/reasoning cards, without disrupting the existing free vs paid gating logic.
6) Preserve current behavior
Ensure the following remain unchanged unless market context is actually available:
state detection behavior
current fee-cap override logic
OpenAI response parsing/validation
locked/unlocked result flow
free preview and paid analysis behavior
7) Verify end to end
After implementation:
run the app locally
confirm /api/analyze still works with no market context available
confirm the prompt includes MARKET_INTELLIGENCE only when warehouse data is returned
confirm the frontend card appears only when marketContext is present
confirm existing analysis flows still work
run the test suite and fix any regressions
