Update the existing Odigos Next.js app to add Stripe monetization. Do NOT rebuild the app.

Goal:
Monetize immediately by paywalling “Suggested Reply” + “Analysis Reasoning” behind a one-time Stripe Checkout purchase.
No auth, no database, no webhook required.

UX requirements:
1) User runs analysis as normal.
2) Results page shows these sections for free:
   - Deal Score + VerdictLabel + Confidence
   - What This Deal Likely Means
   - What We Detected (structured fields)
3) These sections must be LOCKED until paid:
   - Suggested Reply to Dealer
   - Analysis Reasoning
   Show a lock state with a button: “Unlock for $79”

Pricing:
- One-time purchase price: $79 (configurable via Stripe Price ID)

Stripe implementation (simple + secure, no webhooks):
A) Add environment variables:
- STRIPE_SECRET_KEY
- STRIPE_PRICE_ID   (Price ID for $79 product)
- NEXT_PUBLIC_APP_URL (e.g. https://<your-replit-domain>)
B) Create an API route: POST /api/create-checkout-session
- Input: { analysisId: string }
- Create a Stripe Checkout Session with:
  - mode: "payment"
  - line_items: [{ price: STRIPE_PRICE_ID, quantity: 1 }]
  - success_url: `${NEXT_PUBLIC_APP_URL}/results/${analysisId}?session_id={CHECKOUT_SESSION_ID}`
  - cancel_url: `${NEXT_PUBLIC_APP_URL}/results/${analysisId}?canceled=1`
- Return: { url: session.url }
C) On the results page:
- If query param session_id is present:
  - Call a server endpoint to verify it (see D)
  - If valid and paid, unlock the gated sections
- If no session_id (or not paid), keep locked state and show “Unlock for $79” button

D) Create an API route: GET /api/verify-session?session_id=...
- Server-side only using STRIPE_SECRET_KEY
- Retrieve the checkout session from Stripe
- Consider “paid” if:
  - session.payment_status === "paid"
- Return JSON: { paid: boolean }
- Never expose secret keys in client code

E) Important security rule:
- Only unlock if verify-session returns paid=true.
- Do NOT unlock based purely on the presence of a session_id in the URL.
- If verification fails, show a friendly error state and keep locked.

UI details:
- In locked sections, show blurred preview text or a short teaser, e.g.:
  - “Unlock to get a copy/paste reply and reasoning tailored to your deal.”
- The Unlock button should call /api/create-checkout-session and redirect the browser to the returned session URL.

Data handling:
- Keep your existing analysis storage method (in-memory or file-based) as-is.
- The paywall must work without adding a database.
- Use analysisId already present in results routing.

Deliverables:
- Working checkout flow in Replit
- Clear README section: “How to set up Stripe”
- Add a simple “Test purchase” note using Stripe test mode
- Ensure the app still works when Stripe env vars are missing:
  - If STRIPE_SECRET_KEY is not set, hide the unlock button and show “Payments not configured”

Do not add:
- subscriptions
- webhooks
- user accounts
- complex entitlements
This is an MVP paywall only.
