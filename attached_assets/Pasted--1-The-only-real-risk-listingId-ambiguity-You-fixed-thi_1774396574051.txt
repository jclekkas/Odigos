🔴 1. The only real risk: listingId ambiguity

The requirement:
“listingId (the persisted submission row ID)”
The problem:
You’re mixing naming domains:
“listing” (product/UI language)
“submission” (actual DB row)
If this is even slightly unclear in code, you will:
attach feedback to the wrong table
or fail FK constraints later
✅ Fix (do this explicitly)
Make it unambiguous everywhere:
In DB: dealer_submissions.id
In API: keep listingId (fine for product language)
In code comments:
// listingId === dealer_submissions.id (source of truth)
If you want to be even safer:
submissionId: number; // internal
listingId: number;    // API alias (same value)
But minimum: document it clearly once.
