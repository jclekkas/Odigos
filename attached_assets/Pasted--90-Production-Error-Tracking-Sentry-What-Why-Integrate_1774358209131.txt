#90 - Production Error Tracking (Sentry)

What & Why
Integrate Sentry into both the frontend and backend so real production failures are visible immediately after deploy. Odigos is a trust-sensitive consumer product — silent failures in analysis, upload handling, or checkout are unacceptable. This closes the observability loop after Task #88 (tests) and Task #89 (CI).

Done looks like
- Frontend runtime errors, uncaught exceptions, and unhandled promise rejections are captured in Sentry in production
- A React Error Boundary wraps the app so rendering failures are captured and reported
- Backend route errors, unhandled rejections, and uncaught exceptions are captured in Sentry in production
- Critical flows (analyze, extract-text, checkout, state-fee lookup) include lightweight Sentry context (feature area, route, status)
- Errors include useful metadata but NO raw dealer text, document content, uploaded file data, or PII
- Development mode is quiet by default (no Sentry noise locally unless explicitly enabled)
- Missing or unset DSN does NOT crash frontend or backend
- Required environment variables (SENTRY_DSN, VITE_SENTRY_DSN) are documented and added to Replit Secrets
- Integration is documented in replit.md with clear setup + verification steps

Out of scope
- Full logging platform migration
- Metrics dashboards or full distributed tracing
- Session replay
- Alerting/on-call workflows
- Product analytics or event tracking
- Refactoring unrelated production code

Observability Principles (CRITICAL)
- Capture real errors, not everything
- Prioritize debuggability over volume
- Keep integration lightweight and maintainable
- Never leak sensitive user data
- Avoid noisy or duplicate reporting

Privacy & Safety Rules (CRITICAL)
- NEVER send:
  - raw dealer text
  - extracted PDF/image text
  - uploaded file contents
  - API keys, tokens, or secrets
  - unnecessary PII
- Sanitize request data before attaching to Sentry
- Use beforeSend to strip or redact any sensitive fields
- Prefer tags and small metadata over full payloads

Tasks

1. Install Sentry SDKs
Install:
- @sentry/node for backend
- @sentry/react for frontend

Ensure versions are compatible with the current Express + React + Vite setup.
Do not add unnecessary Sentry packages.

2. Backend initialization
Initialize Sentry in server/index.ts BEFORE routes are registered.

Requirements:
- Use Sentry.init with DSN from process.env.SENTRY_DSN
- Guard initialization so missing DSN = no-op (no crash)
- Attach Sentry request handler and error handler to Express
- Use Sentry.setupExpressErrorHandler(app) if compatible, otherwise equivalent middleware
- Capture:
  - route-level exceptions
  - unhandled promise rejections
  - uncaught exceptions

Do not change existing business logic.

3. Frontend initialization
Initialize Sentry in client/src/main.tsx using VITE_SENTRY_DSN.

Requirements:
- Only initialize in production OR when explicitly enabled via env flag
- Enable BrowserTracing for basic navigation context (no heavy config)
- Guard initialization so missing DSN = no-op
- Do not introduce performance-heavy integrations

4. React Error Boundary
Wrap the app with Sentry.ErrorBoundary in client/src/App.tsx or main.tsx.

Requirements:
- Minimal fallback UI (simple message, no redesign)
- No additional UX complexity
- This is the only new React component introduced

5. Add privacy-safe context for critical flows
Add lightweight tagging/scoping around key flows:

Frontend + backend:
- analyze
- extract-text
- checkout (if present)
- state-fee lookup

Include:
- feature area (e.g. "analyze", "extract-text")
- route/path
- status code or error type

Do NOT include:
- raw request bodies
- extracted content
- user-submitted text
- uploaded file data

Use Sentry scope or tags — keep it minimal and useful.

6. Sanitize data via beforeSend
Implement a beforeSend hook:

- Remove or redact:
  - request bodies
  - file contents
  - large payload fields
  - known sensitive keys
- Ensure any accidental sensitive data is stripped before sending

Keep this logic simple and explicit.

7. Environment handling
Ensure correct behavior by environment:

- production:
  - Sentry enabled if DSN is set
- development:
  - Sentry disabled by default OR opt-in via env flag
- missing DSN:
  - safe no-op, no crash

Environment variables:
- backend: SENTRY_DSN
- frontend: VITE_SENTRY_DSN

Do not hardcode values.

8. Minimal verification path
Add a safe way to verify integration:

- optional small test trigger (e.g. manual error throw in dev only or documented snippet)
- or clear instructions for verifying via real error

Do NOT leave unsafe debug endpoints in production.
