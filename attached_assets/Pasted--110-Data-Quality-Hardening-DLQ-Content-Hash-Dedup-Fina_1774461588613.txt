#110 - Data Quality Hardening: DLQ, Content-Hash Dedup, Financial Bounds

Implement this task EXACTLY as specified. Do NOT introduce unrelated refactors. Keep changes minimal, surgical, and consistent with existing patterns.

What & Why
Fix 3 data quality gaps in the warehouse ingestion pipeline:

Warehouse write failures are currently silently dropped
Identical documents can be stored as separate listings
Extracted financial values are not validated for plausibility

Done looks like
SHA-256 content-hash deduplication implemented
Failed warehouse writes captured in an in-DB dead-letter queue (DLQ)
Retry mechanism (max 3 attempts with exponential backoff)
Financial sanity validation implemented with structured flags
Existing behavior preserved (no breaking changes)
Doc-fee cap rule engine remains unchanged

Out of scope
External queue infrastructure (SQS, RabbitMQ, etc.)
Fuzzy duplicate detection
Admin UI for replaying DLQ entries
Schema changes beyond specified fields
Unrelated refactors

Tasks

1. Content-hash dedup

At ingest time (server/ingestor.ts):
- Normalize raw submission text: convert CRLF → LF, trim outer whitespace, do NOT lowercase, do NOT collapse internal whitespace
- Compute SHA-256 hex hash
- Store as content_hash on dealer_submissions

In warehouse (core.listings), add columns:
- content_hash TEXT
- is_duplicate BOOLEAN NOT NULL DEFAULT false
- duplicate_of_listing_id UUID NULL

In warehouseWriter, before inserting a listing:
- Query for existing listing where content_hash matches AND is_duplicate = false
- If found: insert new row with is_duplicate = true, duplicate_of_listing_id = original.id, still linked to current submission — do NOT skip the insert
- If not found: insert normally with is_duplicate = false

Rules:
- Exact-match only
- Same content across different users = duplicate
- If earlier write failed and no original exists, later submission becomes original (acceptable)

2. In-DB dead-letter queue (DLQ)

In shared/schema.ts, create table failed_warehouse_writes:
- id
- submission_id
- payload (JSONB)
- error_message (TEXT)
- attempt_count (INT)
- failed_at (TIMESTAMP)

Indexes:
- submission_id
- failed_at DESC

In warehouseWriter, wrap write logic in retry loop:
- Max 3 attempts
- Backoff:
  - attempt 1 immediate
  - attempt 2 ~100ms
  - attempt 3 ~300ms

Behavior:
- On success → return normally
- On final failure → insert ONE DLQ row with:
  - submission_id
  - safely serialized payload (NO raw document text)
  - error_message
  - attempt_count = 3
- Do NOT silently swallow errors anymore

3. Financial sanity bounds

Bounds map:
- vehiclePrice: 500–500000
- tradeInValue: 0–500000
- docFee: 0–10000
- dealerAddonCost: 0–100000
- totalFees: 0–10000

Rules:
- Run BEFORE warehouse insert
- Do NOT reject write
- Store flags in core.listings.sanity_flags JSONB NOT NULL DEFAULT []
- Flag format:
  { field: string, value: number, min?: number, max?: number, reason: "below_min" | "above_max" | "not_finite" }
- Missing values → no flag
- NaN/Infinity → reason = "not_finite"
- This is NOT part of rule engine scoring

4. Helpers (server/warehouse/warehouseUtils.ts)

Add small typed helpers:
- normalizeSubmissionText(raw: string): string
- sha256Hex(input: string): string
- delay(ms: number): Promise<void>
- getBackoffMs(attempt: number): number
- validateFinancialBounds(...): SanityFlag[]
- safeSerializePayload(...): object
- getErrorMessage(...): string

5. Tests (required)

Add/update tests for:
- Same normalized text → identical SHA-256
- CRLF vs LF normalization
- First submission → non-duplicate listing inserted
- Second identical submission → duplicate row linked to original
- Retry succeeds on transient failure before DLQ insertion
- Persistent failure → exactly ONE DLQ row with attempt_count = 3
- Out-of-bounds financial values still insert the listing and populate sanity_flags
- Existing doc-fee cap logic unchanged
