#89 — Implement PII deletion job
What & Why
Odigos has a stated 90-day TTL policy for personally identifiable information (PII), but it is not enforced. This means user-submitted data can persist indefinitely, creating unnecessary privacy and compliance risk.
This task enforces automatic deletion/redaction of PII after 90 days. This is a data minimization control, not a product feature.
Goal
Automatically delete or irreversibly redact PII-bearing records older than 90 days via a safe, auditable, idempotent background job.
Done looks like
A scheduled server-side job runs automatically and enforces a 90-day retention window
All PII-bearing tables are covered (no gaps)
A single canonical timestamp (createdAt) is used for cutoff logic
Job is idempotent (safe to run multiple times)
Job logs include:
start time
end time
cutoff timestamp
records scanned per table
records deleted/redacted per table
any failures
Dry-run mode exists (no data mutation)
Manual trigger exists (npm run pii:cleanup)
No PII persists beyond 90 days unless explicitly justified in code
README or script comment explains how it works and how to run it
Out of scope
Redesigning the data model
Building a user-facing privacy portal
Reprocessing historical analytics
Deleting non-PII aggregated metrics
Any unrelated product or UI changes
Tasks
1. Inventory all PII
Identify every table/column that stores user-submitted or identifying data, including:
raw quote text
uploaded/extracted text
pasted messages/emails
names, emails, phone numbers, addresses
file metadata if persisted
logs containing raw payloads
analysis outputs that echo user input
Create a short internal note listing:
tables in scope
why each contains PII
delete vs redact decision
2. Define retention rule
Use a single constant:
PII_RETENTION_DAYS = 90
Compute cutoff:
cutoff = now - 90 days
Use createdAt as the canonical field. Do NOT mix timestamps.
3. Deletion vs redaction
Hard delete: raw submissions and any primary PII storage
Redact: records that must persist (analytics, financial linkage)
Redaction must remove all user-identifiable content and leave only minimal non-identifying metadata.
Default rule: delete unless there is a strong reason not to
4. Implement cleanup job
Create:
server/jobs/piiCleanup.ts
Requirements:
batch processing (no large in-memory loads)
per-table execution
safe retries
structured logging
supports dryRun flag
configurable batch size
5. Manual execution
Create:
server/scripts/runPiiCleanup.ts
Expose:
npm run pii:cleanup
Supports:
normal mode
dry-run mode
6. Scheduling
Hook into existing background job / cron system.
Requirements:
runs automatically (daily is sufficient)
does not block server startup
failures do not crash the app
7. Logging (required)
Emit structured logs:
pii_cleanup_started
pii_cleanup_table_scanned
pii_cleanup_table_deleted
pii_cleanup_completed
pii_cleanup_failed
Each log must include:
cutoff timestamp
table name
counts
8. Downstream coverage
Ensure PII is not retained in:
warehouse (raw/core tables)
derived analysis tables
logs
Do NOT assume only the primary DB contains PII.
9. Tests
Add tests for:
records < 90 days are preserved
records > 90 days are deleted/redacted
exact 90-day boundary behavior
dry-run makes no changes
idempotency (run twice = same result)
partial failures are logged and isolated
Suggested structure
server/jobs/piiCleanup.ts
server/scripts/runPiiCleanup.ts
shared retention config
test coverage per table
Constraints (non-negotiable)
Do NOT change pricing
Do NOT change analyzer logic
Do NOT change routing or UI
Do NOT add accounts or dashboards
Do NOT store additional data “to help later”
Do NOT rely on policy — enforcement must be in code
Acceptance report (required)
At completion, output:
all tables touched
delete vs redact decision per table
exact cutoff logic
scheduling mechanism
sample dry-run output
any exclusions + justification
