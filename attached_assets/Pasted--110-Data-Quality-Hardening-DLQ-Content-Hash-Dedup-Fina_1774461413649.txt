#110 - Data Quality Hardening: DLQ, Content-Hash Dedup, Financial Bounds

Implement this task EXACTLY as specified. Do NOT introduce unrelated refactors. Keep changes minimal, surgical, and consistent with existing patterns.

--------------------------------
GOAL
--------------------------------
Fix 3 data quality gaps in the warehouse ingestion pipeline:

1) Warehouse write failures are currently silently dropped
2) Identical documents can be stored as separate listings
3) Extracted financial values are not validated for plausibility

--------------------------------
REQUIRED OUTCOMES
--------------------------------
- SHA-256 content-hash deduplication implemented
- Failed warehouse writes captured in an in-DB dead-letter queue (DLQ)
- Retry mechanism (max 3 attempts with exponential backoff)
- Financial sanity validation implemented with structured flags
- Existing behavior preserved (no breaking changes)
- Doc-fee cap rule engine remains unchanged

--------------------------------
IMPLEMENTATION DETAILS
--------------------------------

========================
1) CONTENT-HASH DEDUP
========================

At ingest time (server/ingestor.ts):

- Normalize raw submission text:
  - Convert CRLF → LF
  - Trim outer whitespace
  - DO NOT lowercase
  - DO NOT collapse internal whitespace

- Compute SHA-256 hex hash
- Store as `content_hash` on dealer_submissions

In warehouse (core.listings):

- Add columns:
  - content_hash TEXT
  - is_duplicate BOOLEAN NOT NULL DEFAULT false
  - duplicate_of_listing_id UUID NULL

In warehouseWriter:

- Before inserting listing:
  - Query for existing listing where:
    - content_hash matches
    - is_duplicate = false

- If found:
  - Insert new row with:
    - is_duplicate = true
    - duplicate_of_listing_id = original.id
    - still link to current submission
  - DO NOT skip insert

- If not found:
  - Insert normally with is_duplicate = false

Important:
- Dedup is exact-match only
- Same content across users = duplicate
- If earlier write failed and no original exists, later one becomes original (acceptable)

========================
2) DEAD-LETTER QUEUE (DLQ)
========================

In shared/schema.ts:

Create table:
- failed_warehouse_writes
  - id
  - submission_id
  - payload JSONB
  - error_message TEXT
  - attempt_count INT
  - failed_at TIMESTAMP

Indexes:
- submission_id
- failed_at DESC

In warehouseWriter:

- Wrap write logic in retry loop:
  - Max 3 attempts
  - Backoff example:
    - attempt 1: immediate
    - attempt 2: ~100ms
    - attempt 3: ~300ms

- On success → return normally

- On final failure:
  - Insert ONE DLQ row with:
    - submission_id
    - SAFE serialized payload (NO raw document text)
    - error_message
    - attempt_count = 3

- DO NOT silently swallow errors anymore

========================
3) FINANCIAL SANITY BOUNDS
========================

Define bounds map:

vehiclePrice: 500 → 500000  
tradeInValue: 0 → 500000  
docFee: 0 → 10000  
dealerAddonCost: 0 → 100000  
totalFees: 0 → 10000  

Validation rules:

- Run BEFORE warehouse insert
- DO NOT reject write
- Instead, store flags in:

core.listings.sanity_flags JSONB NOT NULL DEFAULT []

Flag format:
[
  {
    field: string,
    value: number,
    min?: number,
    max?: number,
    reason: "below_min" | "above_max" | "not_finite"
  }
]

Rules:
- Missing values → no flag
- NaN / Infinity → reason = "not_finite"
- This is NOT part of rule engine scoring
