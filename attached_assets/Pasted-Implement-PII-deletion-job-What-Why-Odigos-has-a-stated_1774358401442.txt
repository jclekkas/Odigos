Implement PII deletion job
What & Why
Odigos has a stated 90-day TTL policy for personally identifiable information, but it is not actually enforced. That means PII can persist indefinitely unless manually removed. This creates unnecessary privacy and compliance risk. California’s CCPA gives consumers a right to request deletion of personal information, subject to exceptions, and California privacy rules emphasize retaining personal information only as reasonably necessary and proportionate for disclosed purposes. A working TTL-based deletion job is a practical data-minimization control and should be enforced in code, not policy docs alone.

Goal
Automatically delete or irreversibly redact PII-bearing records once they are older than 90 days, with a safe, auditable, idempotent background job.

Done looks like

A scheduled server-side deletion job runs automatically and enforces the 90-day retention window.
Any table/record that stores user-submitted analysis input or other PII is covered.
Deletion is based on a single canonical timestamp field, not ad hoc logic.
The job is idempotent: running it twice does not fail or produce inconsistent results.
The job logs:
start time
end time
cutoff timestamp used
records scanned
records deleted or redacted by table
any failures
There is a dry-run mode for verification in dev/staging.
There is a manual trigger script/endpoint for operators to run it on demand.
The app no longer retains PII-bearing data past 90 days unless a narrowly defined legal/operational exception is explicitly documented in code.
README or ops notes explain how the job works, what it deletes, and how to run dry-run/manual execution.
Out of scope
Redesigning the data model
Building a full privacy request portal
Reprocessing historical analytics
Deleting non-PII aggregated warehouse metrics that are no longer user-identifiable
Changing product behavior unrelated to retention
Implementation requirements
Inventory PII-bearing data first
Identify every table/column storing user-submitted content or directly identifying data, including things like:
raw quote text
uploaded extracted text
email/message text pasted by users
names, phone numbers, email addresses, mailing addresses if present
attachments or file metadata if persisted
request logs containing raw user payloads
analysis artifacts that echo user-submitted text
Produce a short internal note in the repo describing:
which tables are in scope
why each is in scope
whether each should be deleted vs redacted
Use a canonical retention rule
Define one retention constant:
PII_RETENTION_DAYS = 90
Compute a single cutoff:
now - 90 days
Use a single canonical timestamp per record, ideally createdAt. Do not mix updatedAt unless there is a very specific documented reason.
Prefer hard delete for raw PII
For raw submissions and similar PII-heavy records, hard delete them once expired.
For records that must remain for product analytics or financial reconciliation:

remove/redact the PII fields only
preserve only the minimum non-identifying metadata genuinely needed
The principle should be: keep the least data necessary. That aligns with modern privacy expectations and California’s proportionality/necessity framing.
Make the job safe and auditable
Build the deletion job so it:
runs in batches
does not load huge result sets into memory
can resume safely after failure
uses structured logs
surfaces errors clearly
Include:
dryRun: true/false
batch size config
per-table counters
Provide both scheduled and manual execution
Add:
a scheduled execution path
a manual script such as npm run pii:cleanup
The manual command should support:
normal mode
dry-run mode
Protect downstream systems
If PII is copied into warehouse/raw/core tables, logs, or derived analysis tables, ensure the job also covers those copies or confirm they never contain user-identifiable text. Do not assume the application DB is the only place PII lives.
Do not silently delete without visibility
Emit logs like:
pii_cleanup_started
pii_cleanup_table_scanned
pii_cleanup_table_deleted
pii_cleanup_completed
pii_cleanup_failed
Logs must include the cutoff date and counts.
Test coverage required
Add tests for:
records newer than 90 days are preserved
records older than 90 days are deleted/redacted
exactly-on-threshold behavior is deterministic
dry-run deletes nothing
repeated execution is safe
failures in one batch/table are logged properly and do not corrupt other work
Suggested implementation shape
server/jobs/piiCleanup.ts — core cleanup logic
server/scripts/runPiiCleanup.ts — manual execution entrypoint
scheduler hook in existing cron/background job mechanism
shared config constant for retention days
tests covering each scoped table
Important product/engineering constraints
Do not change pricing, routing, or user-facing flows
Do not add accounts or a privacy dashboard
Do not store extra data to “help” with deletion later
Do not rely on policy text as enforcement
Do not delete payment/accounting records that must legally be retained, but do strip any unnecessary PII from adjacent product records where possible
Acceptance note
At the end, report:
every table touched
delete vs redact decision for each
exact retention cutoff logic
how scheduling works
dry-run output example
any records intentionally excluded and why
One nuance: this is a strong privacy/compliance improvement, but it is not by itself a complete legal compliance program. It materially reduces risk because it enforces retention minimization and supports deletion expectations under laws like CCPA, but legal review is still appropriate for any exceptions or statutory retention obligations.
