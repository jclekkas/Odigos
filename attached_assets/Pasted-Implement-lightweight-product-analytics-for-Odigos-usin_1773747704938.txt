Implement lightweight product analytics for Odigos using PostHog.

This is a surgical instrumentation task.

DO NOT change:
- backend logic
- pricing
- routing behavior
- analysis prompt
- API payloads
- result rendering
- visual design beyond what is required for initialization

OBJECTIVE

Add decision-grade funnel tracking so we can measure:
1) landing page CTA clicks
2) analyzer page views
3) input started
4) analysis submitted
5) analysis completed
6) paid conversion (only if there is already an existing payment success point)

We need clean event tracking only.
Do NOT track pasted dealer text, uploaded file contents, names, emails, phone numbers, or any user-entered quote content.

--------------------------------
TECH STACK / IMPLEMENTATION EXPECTATIONS

- Use PostHog
- Keep implementation simple and production-safe
- Use env var(s) for PostHog key / host
- Initialize once at app startup
- Do not break SSR/build if env vars are missing
- If PostHog env vars are absent, fail gracefully and no-op

--------------------------------
RELEVANT EVENTS TO ADD

1. landing_cta_clicked
Fire when a user clicks the main homepage CTA that takes them to /analyze

Properties:
- location: "hero"
- cta_text

2. analyze_page_viewed
Fire when the /analyze page loads

Properties:
- route: "/analyze"

3. analysis_input_started
Fire once per page load/session when the user first types OR pastes into the main analyzer textarea
IMPORTANT:
- only fire once per visit to the page
- do not fire repeatedly on every keystroke

Properties:
- input_method: "typing" or "paste"

4. analysis_submitted
Fire when the user submits an analysis request

Properties:
- input_mode: "paste" or "upload" if that distinction already exists in UI state
- has_optional_details: true/false if easily available without adding logic
- input_length_bucket: one of ["1-99","100-299","300-599","600+"]

IMPORTANT:
- bucket only
- do NOT send actual text
- do NOT send raw character count if avoidable; bucket is preferred

5. analysis_completed
Fire when analysis successfully returns and the user sees the result state

Properties:
- result_available: true

6. paid_conversion
ONLY add this if there is already a clear existing client-side success point for completed payment / unlock.
If present, fire there.

Properties:
- product: "full_review"

Do NOT invent a payment success flow if one does not already exist.
If paid conversion hook location is ambiguous, leave it untouched and note where it should be added later.

--------------------------------
IMPLEMENTATION DETAILS

1. Add a tiny analytics utility
Create a small helper module for:
- safe PostHog initialization
- capture(eventName, properties)
- no-op behavior when env vars are missing

2. Initialize PostHog once
Place initialization at app startup in the appropriate client entry point / root app layer.

3. Instrument existing UI only
Do not restructure components just to add analytics.
Add event calls at existing interaction points.

4. Input-start handling
For the main analyzer textarea:
- detect first keyboard entry -> fire analysis_input_started once
- detect first paste -> fire analysis_input_started once with input_method="paste"
- once fired, do not fire again until page reload/navigation away and back

5. Submission handling
Instrument the existing submit action only.
Do not modify submission logic.

6. Completion handling
Instrument the point where analysis has successfully completed and the result is shown.
Do not fire on loading.
Do not fire on failed requests.
