#90 - Production Error Tracking (Sentry)

What & Why
Integrate Sentry into both the frontend and backend so real production failures are visible immediately after deploy. Odigos is a trust-sensitive consumer product — silent failures in analysis, upload handling, or checkout are unacceptable. This completes the reliability loop after Task #88 (tests) and Task #89 (CI).

---

Done looks like
- Frontend runtime errors, uncaught exceptions, and unhandled promise rejections are captured in Sentry in production
- A React Error Boundary wraps the app and captures render failures
- Backend route errors, unhandled rejections, and uncaught exceptions are captured in Sentry in production
- Critical flows (analyze, extract-text, checkout, state-fee lookup) include lightweight context (feature, route, status)
- Errors include useful metadata but NO raw dealer text, document content, uploaded file data, or PII
- Development mode is quiet by default (no Sentry noise unless explicitly enabled)
- Missing or unset DSN does NOT crash frontend or backend
- Required environment variables (SENTRY_DSN, VITE_SENTRY_DSN) are documented and added to Replit Secrets
- Integration is documented clearly in replit.md with setup and verification steps

---

Out of scope
- Full logging platform migration
- Metrics dashboards or distributed tracing
- Session replay
- Alerting/on-call workflows
- Product analytics/event tracking
- Refactoring unrelated production code

---

Observability Principles (CRITICAL)
- Capture real, unexpected failures only
- Optimize for debuggability over volume
- Keep the system lightweight and maintainable
- Avoid noisy or duplicate reporting
- Never leak sensitive user data

---

Privacy & Safety Rules (CRITICAL)
- NEVER send:
  - raw dealer text
  - extracted PDF/image text
  - uploaded file contents
  - API keys, tokens, or secrets
  - unnecessary PII
- Sanitize all request data before sending
- Use beforeSend to strip or redact sensitive fields
- Prefer tags/metadata over full payload capture

---

Noise Control Rule
- Do NOT capture handled or expected errors
- Avoid duplicate reporting of the same error
- Capture only actionable failures
- Keep event volume high-signal

---

Duplicate Reporting Guard
- Avoid sending the same error from both frontend and backend
- Backend = source of truth for server failures
- Frontend = source of truth for UI/rendering failures
- Do not suppress legitimate distinct errors

---

Error Boundary Rule
- Error Boundary captures render failures only
- Do NOT use it as a catch-all for normal errors
- Preserve existing error handling flows

---

Simplicity Rule
- Use the simplest implementation that provides real visibility
- Avoid unnecessary abstractions or wrappers
- Prefer direct integration over complexity

---

Refactor Guardrail
- Do NOT refactor unrelated production code
- Only make minimal, necessary changes
- If a bug is found, fix it and document it

---

Tasks

1. Install Sentry SDKs
Install:
- @sentry/node (backend)
- @sentry/react (frontend)

Ensure compatibility with Express + React + Vite.
Do NOT add unnecessary Sentry packages.

---

2. Backend initialization
Initialize in server/index.ts BEFORE routes.

Requirements:
- Sentry.init using process.env.SENTRY_DSN
- Missing DSN = safe no-op (no crash)
- Attach request + error handlers to Express
- Capture:
  - route exceptions
  - unhandled promise rejections
  - uncaught exceptions
- Do NOT modify business logic

---

3. Frontend initialization
Initialize in client/src/main.tsx using VITE_SENTRY_DSN.

Requirements:
- Enable only in production or explicit opt-in
- Enable BrowserTracing (basic only)
- Missing DSN = safe no-op
- Avoid heavy performance integrations

---

4. React Error Boundary
Wrap app with Sentry.ErrorBoundary in App.tsx or main.tsx.

Requirements:
- Minimal fallback UI (simple message)
- No UX redesign
- Only new component introduced

---

5. Privacy-safe context for critical flows
Add lightweight tagging for:
- analyze
- extract-text
- checkout (if present)
- state-fee lookup

Include:
- feature name
- route/path
- status code or error type

Do NOT include:
- request bodies
- extracted content
- user text
- uploaded files

---

6. Sanitize via beforeSend
Implement beforeSend (frontend + backend):

- Remove/redact:
  - request bodies
  - file contents
  - large payload fields
  - sensitive keys
- Keep logic simple and explicit

---

7. Environment handling
- production → Sentry enabled if DSN set
- development → disabled by default (opt-in only)
- missing DSN → safe no-op

Env vars:
- backend: SENTRY_DSN
- frontend: VITE_SENTRY_DSN

No hardcoded values.

---

8. Minimal verification path
Provide a safe way to verify integration:

- documented manual test OR controlled dev-only error trigger
- no unsafe debug endpoints in production

---

9. Documentation (replit.md)
Add section covering:
- Sentry integration
- initialization points (frontend + backend)
- required env vars (Replit Secrets)
- what is captured
- what is excluded (privacy)
- how to verify

Do NOT create new docs.
