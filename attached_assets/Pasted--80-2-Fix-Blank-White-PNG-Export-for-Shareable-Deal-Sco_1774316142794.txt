#80.2 - Fix Blank/White PNG Export for Shareable Deal Scorecard

What & Why
The Shareable Deal Scorecard feature is implemented, but in development the downloaded PNG is rendering as a blank/white image instead of the actual scorecard content. This is not intended behavior.

The likely problem is in the html-to-image capture pipeline: the hidden ShareCard node is either being positioned in a way that prevents reliable capture, being captured before styles/layout/fonts are fully ready, or not being rendered in a capture-safe way.

This task is a surgical bug fix. Do not redesign the feature or expand scope.

Done looks like
- Clicking "Download scorecard" downloads a PNG that contains the fully rendered scorecard, not a blank/white image
- The PNG includes the Odigos branding, verdict badge, verdict label, issue list, and footer
- Export works reliably in dev in Chrome, Firefox, and Safari
- The hidden ShareCard remains invisible to users in normal UI, but is still capture-safe for html-to-image
- No changes to feature scope, placement, or backend

Out of scope
- Redesigning the scorecard
- Changing the copy/share feature set
- Adding backend persistence
- Adding new dependencies
- Changing analytics, routing, pricing, or paywall logic

Tasks

1) Make ShareCard capture-safe
File:
- client/src/components/ShareCard.tsx

Update the hidden/off-screen rendering approach so the scorecard remains fully renderable by html-to-image.

Requirements:
- Do NOT use display: none
- Do NOT rely on a positioning approach that may cause capture to fail
- Prefer a capture-safe off-screen strategy such as:
  - position: fixed
  - top: 0
  - left: 0
  - transform: translate(-200vw, -200vh)

Ensure the root card element has explicit inline or reliable styles for:
- width: 1200px
- height: 630px
- solid white background
- visible text color
- consistent padding

Do not allow parent overflow/clipping to wipe the node out.

2) Harden capture timing
File:
- client/src/pages/home.tsx

Before calling html-to-image export:
- ensure the ShareCard node exists
- wait for layout/render to settle before capture

Use a stronger capture delay:
- at least two requestAnimationFrame waits
- plus a short timeout if needed (for example ~50ms)

Goal:
avoid capturing before the node is fully painted/styled.

3) Force background and export options
In the html-to-image call:
- explicitly set backgroundColor: "#ffffff"
- preserve pixelRatio: 2 (or current retina-safe equivalent)
- keep export crisp and readable

Ensure the export is not transparent and does not default to a blank white canvas without content.

4) Add a temporary debug verification path during implementation
During implementation only, verify that the ShareCard node actually contains rendered content at capture time.

You may:
- confirm the node exists before capture
- confirm it has non-empty content
- confirm dimensions are correct

Do not leave noisy console logging behind in final code unless minimal and necessary for error handling.

5) Fail gracefully if capture node is missing
If the ShareCard node is not found or export fails:
- do not silently download a bad file
- provide lightweight user feedback
- keep existing feature behavior otherwise unchanged

6) Validate final behavior
Confirm the exported PNG:
- is not blank
- contains all intended visible content
- looks correct in dev
- works in Chrome, Firefox, and Safari

Relevant files
- client/src/components/ShareCard.tsx
- client/src/pages/home.tsx

Deliverables
Return:
1. Exact files changed
2. Root cause found
3. Specific fix applied to hidden rendering/capture timing
4. Confirmation that the PNG now contains the rendered scorecard content
5. Confirmation that no feature scope changed

End of task
