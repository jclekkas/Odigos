#87 - Fix Warehouse Schema in Production + Restart Dev

What & Why
The production database is missing the entire warehouse schema: both raw and core schemas, all warehouse tables, and all materialized views. Every user analysis since launch has silently failed to write to the warehouse.

Root cause:
scripts/post-merge.sh runs db:push interactively and aborts when it detects the materialized views would be dropped, so the schema push never completes in production.

The dev server also needs a restart after a port conflict from the last merge.

Objective:
1. Make warehouse schema creation reliable in production
2. Prevent future deploys from aborting on the db:push prompt
3. Add a safe startup-time warehouse bootstrap
4. Restart dev and confirm preview is running again

1) Fix scripts/post-merge.sh for non-interactive schema push

File:
- scripts/post-merge.sh

Problem:
db:push aborts when it detects that materialized views would be dropped.

Required behavior:
- Before running db:push, explicitly drop these materialized views if they exist:
  - core.platform_metrics
  - core.national_stats
  - core.state_stats
- Use schema-qualified names
- Then run db:push non-interactively using:
  - --accept-data-loss
  OR an equivalent fully non-interactive approach
- After db:push succeeds, immediately re-run the warehouse view setup so the materialized views are recreated

Requirements:
- Script must be fully non-interactive
- Script must be repeatable and production-safe
- Script must fail clearly on any real error
- Recreating views must happen only after a successful db:push
- Do not swallow db:push failures unrelated to the views

Implementation notes:
- Preserve shell safety if already present (set -euo pipefail or equivalent)
- Use explicit DROP MATERIALIZED VIEW IF EXISTS statements
- Keep the flow deterministic:
  1. drop views
  2. push schema
  3. recreate views

Requirements:

- Must be safe to run on every startup
- Must reuse existing warehouse setup logic and shared definitions where possible
- Must be wrapped in try/catch
- Must log clearly whether bootstrap was skipped, executed, or failed
- Must never crash the server on failure
- Server startup must continue even if bootstrap fails
