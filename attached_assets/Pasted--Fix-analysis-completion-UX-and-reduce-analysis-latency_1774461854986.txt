# Fix analysis completion UX and reduce analysis latency

Implement this task EXACTLY as specified. Keep changes minimal, surgical, and consistent with existing patterns. Do NOT introduce unrelated refactors.

## What & Why
Two UX failures undermine trust in the analysis flow:

1. Misleading time copy
The form tells users results arrive “in seconds” and a CTA component says “about 10 seconds,” but real latency is often 40+ seconds. This creates false expectations and makes users think the app is broken.

2. Silent completion with no scroll
When analysis finishes, the loading button quietly reverts to “Analyze Deal” while results render below the fold. Users who haven’t scrolled down don’t realize the analysis succeeded and assume the action failed.

But the bigger issue is the latency itself. A trust-sensitive consumer flow cannot take 40+ seconds unless the result is extraordinary. This task therefore includes both:
- a frontend UX trust fix
- a surgical latency reduction pass on the blocking analysis path

## Done looks like
- All misleading analysis wait-time copy is replaced with honest language
- The loading state clearly tells users the analysis is still running and that the wait is normal
- On successful completion, the page automatically scrolls to the results section
- Completion is unmistakable even if the user never manually scrolls
- End-to-end analysis latency is materially reduced from current baseline
- Non-essential blocking work is removed from the critical path where possible
- Existing output quality and business behavior remain unchanged
- No regressions to mobile layout, loading states, or disabled button behavior

## Out of scope
- Full architecture redesign
- Streaming partial results unless trivially supported already
- New background queue infrastructure
- Major redesign of the analyze page
- New dependencies
- Unrelated refactors

## Part 1 — Frontend trust / completion UX

### 1. Replace misleading time copy
Find all user-facing analysis flow copy that implies very fast results, including:
- “in seconds”
- “about 10 seconds”
- any similar wording in analyze page copy, CTA blocks, helper text, submit/loading text, empty states, or hero messaging

Replace with honest copy aligned to current reality without sounding broken or alarming.

Use copy in this spirit:
- “Results usually take about 40–60 seconds”
- “We’ll analyze your deal and show results on this page”
- “This can take up to about a minute”

Do NOT promise very short turnaround.

### 2. Improve in-progress loading state
During analysis submission/loading:
- Keep the button in a clear loading state
- Show nearby status text that explicitly says:
  - the app is actively working
  - the wait is normal
  - the user should stay on the page

Use lightweight native UI only. No heavy redesign.

Suggested direction:
- “Analyzing your deal now… this usually takes 40–60 seconds.”
- “Still working — this is normal and can take up to about a minute.”

### 3. Auto-scroll to results on success
When analysis finishes successfully and valid results are rendered:
- Automatically scroll the results container into view
- Use smooth scrolling
- Trigger only after results are mounted/rendered
- Ensure it works on desktop and mobile
- Do NOT scroll on errors

Implementation guidance:
- Add a ref to the rendered results container
- Detect the transition from loading -> success with valid analysis data
- Call `scrollIntoView({ behavior: "smooth", block: "start" })`

### 4. Make completion unmistakable
On successful completion:
- Ensure success is obvious before or as the page scrolls
- The scroll itself plus the visible results region should make completion clear
- Add a minimal success/status line above results if needed, e.g.:
  - “Your analysis is ready”

Do NOT add:
- toast libraries
- modals
- celebratory animations
- unrelated UI flourishes

## Part 2 — Reduce analysis latency

### 5. Profile the full blocking path
Measure the current end-to-end latency of the analysis request path and identify where time is spent.

Instrument timing around each major stage on the server-side blocking path, such as:
- request receipt
- input normalization / validation
- text extraction or parsing
- state detection / market context fetch
- warehouse/database reads
- LLM call
- post-processing / scoring
- warehouse/database writes
- response serialization

Use existing logging/metrics patterns if present. Do NOT add noisy permanent logging everywhere. Keep instrumentation targeted and useful.

### 6. Identify and reduce the largest bottlenecks
Make a surgical pass to reduce blocking latency without changing the user-visible output quality.

Focus on:
- eliminating unnecessary sequential waits
- parallelizing independent async work
- removing non-essential work from the blocking path
- reducing unnecessary duplicate reads or duplicate computation
- tightening oversized timeouts/retries if they are inflating normal-case latency
- reducing prompt/context bloat only if output quality is preserved

Examples of acceptable changes:
- fetch independent context in parallel instead of serially
- move non-user-visible writes/metrics/bookkeeping off the critical path if safe
- avoid waiting on secondary enrichment before returning the core result
- avoid duplicate DB queries for data already available
- trim redundant prompt sections that do not materially affect output

### 7. Preserve output and behavior
Do NOT change:
- core API contract unless absolutely required and frontend is updated accordingly
- result schema
- score semantics
- fee logic
- business rules
- analysis meaning/content beyond non-material latency optimizations
- security/governance behavior except to move non-essential writes off the response-critical path if safe and behaviorally equivalent

If a side effect is important for correctness, durability, billing, or compliance, do NOT silently remove it. Only move it off the critical path if that preserves required guarantees.

### 8. Prioritize the LLM path
If the LLM call is the dominant bottleneck, optimize around it surgically:
- remove redundant prompt context
- ensure pre-LLM fetches are not serialized unnecessarily
- avoid post-LLM blocking enrichment if it can be computed after the core response or from already-fetched data
- preserve output quality and existing analysis structure

Do NOT swap models or degrade the analysis just to make it faster unless the existing code already supports a clearly equivalent faster configuration.
