90 - Production Error Tracking and Observability (Odigos)

What & Why
Set up production-grade error tracking and basic observability for Odigos so real-world failures are visible immediately after deploy. Task #88 created pre-merge safety via tests. Task #89 adds merge-time enforcement via CI. This task closes the loop by providing post-deploy visibility into production issues.

Odigos is a trust-sensitive consumer product. Silent failures in analysis, upload handling, checkout flow, API routes, or frontend rendering are unacceptable. We need fast visibility into production-only issues that tests may miss.

Primary goal
Integrate a lightweight, credible production error tracking system with enough context to diagnose real failures across both frontend and backend without overengineering.

Default preference
Use Sentry unless the repo already has another error tracking tool partially in place. If another system already exists, inspect first and build on it only if that is clearly the better path.

Done looks like
- Frontend runtime errors are captured in production
- Backend/server exceptions are captured in production
- API route failures are visible with useful error context
- Unhandled promise rejections and uncaught exceptions are captured
- Errors include enough metadata to debug without exposing sensitive user data
- Environment-specific behavior is correct (development vs production)
- Local development is not spammed with noisy reporting
- The integration is documented clearly
- No secrets are hardcoded
- Sensitive user input is not leaked into telemetry

Out of scope
- Full logging platform migration
- Metrics dashboards for every endpoint
- Distributed tracing across multiple services unless trivial
- Analytics/product-event instrumentation
- Session replay unless it is lightweight, clearly beneficial, and privacy-safe
- Alerting/on-call workflow design beyond basic setup guidance

Observability Philosophy
This should be:
- lightweight
- privacy-conscious
- production-useful
- easy to maintain
- enough to catch and diagnose real failures

Do not turn this into a giant telemetry project.

Privacy and Safety Rules (CRITICAL)
Odigos handles sensitive user-submitted dealer quotes/messages and potentially personal information.

Therefore:
- Do NOT send raw dealer text, uploaded document contents, or full user-submitted payloads to error tracking
- Do NOT send API keys, auth secrets, tokens, or payment secrets
- Do NOT send unnecessary PII
- Scrub or redact sensitive fields before reporting
- Prefer high-value context over full raw payload capture

If breadcrumbs, tags, or request context are included, they must be privacy-safe.

Tasks

1. Inspect the repo and identify the integration points
Carefully inspect:
- frontend entry points
- backend/server startup files
- route handling
- existing error boundaries
- existing logger/error handler middleware if any
- environment variable handling

Determine the cleanest minimal integration points for:
- frontend exception capture
- backend exception capture
- API route exception capture
- unhandled process-level failures

2. Add frontend error tracking
Integrate production-safe client-side error tracking.

Capture:
- uncaught exceptions
- unhandled promise rejections
- React runtime/rendering failures where possible
- route/page context if feasible

If the app does not already have a React error boundary, add a minimal one only if it materially improves error capture and is low-risk.

Keep this simple and non-invasive.

3. Add backend/server error tracking
Integrate server-side error tracking for:
- thrown route errors
- unhandled promise rejections
- uncaught exceptions
- unexpected failures in analysis/extraction/payment-related server logic

Ensure the implementation fits the existing server architecture cleanly.

4. Add privacy-safe context
Include useful metadata such as:
- route name/path
- environment
- high-level feature area (analyze, extract-text, checkout, state-fee)
- error code/status where applicable

Do NOT include:
- raw dealer text
- extracted document text
- full uploaded file contents
- secrets
- unnecessary PII

If request context is attached, sanitize it first.

5. Add targeted instrumentation around critical flows
Add lightweight error capture/context around the highest-risk product surfaces:
- analyze flow
- upload/extract-text flow
- checkout/payment initiation flow if present
- state-fee lookup endpoint
- frontend analyzer submission/results rendering

This does NOT mean adding logs everywhere.
Only add instrumentation where it materially improves diagnosis.

6. Environment handling
Ensure behavior is correct by environment:
- production sends errors to the tracking service
- development is quiet or disabled by default unless explicitly enabled
- configuration is driven by environment variables
- missing DSN/config does not crash the app

7. Add minimal validation path
Verify the integration in a safe, testable way.

Examples:
- a controlled test route or temporary validation hook if appropriate
- a clearly documented local verification step
- production-safe initialization checks

Do NOT leave behind unsafe debug endpoints unless clearly gated and intentional.

8. Documentation
Add documentation to replit.md and/or a dedicated section covering:
- what service was integrated
- where it is initialized
- required environment variables
- what kinds of errors are captured
- what is intentionally excluded for privacy
- how to verify the integration works

Simplicity Rule
- Use the simplest implementation that provides meaningful production visibility
- Avoid unnecessary abstractions
- Prefer clear, direct integration over framework-heavy wrappers unless truly justified

Refactor Guardrail
- Do NOT refactor unrelated production code
- Make only minimal, targeted changes required for integration
- If a bug is discovered, fix it and document it clearly

Deliverables (REQUIRED)
When finished, provide:
1. service/tool integrated
2. files created/modified
3. initialization points for frontend and backend
4. required environment variables/secrets
5. what errors are captured
6. what sensitive data is intentionally excluded or sanitized
7. how environment-specific behavior works
8. how to verify the integration
9. any limitations or future observability opportunities

Success criteria
- Production errors become visible quickly
- Critical flows have enough context for diagnosis
- Sensitive user data is protected
- The setup is lightweight, maintainable, and useful on day one
