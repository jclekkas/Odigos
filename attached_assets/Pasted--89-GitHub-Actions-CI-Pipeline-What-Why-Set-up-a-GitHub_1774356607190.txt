#89 - GitHub Actions CI Pipeline

What & Why
Set up a GitHub Actions CI workflow that automatically runs the existing automated test suite from Task #88 on every pull request and every push to main. The goal is to catch regressions before they merge, enforce the current quality bar automatically, and do so without unnecessary complexity or slowdowns.

This task is CI only. It is not CD, deployment orchestration, or infrastructure redesign.

Done looks like
- A valid .github/workflows/ci.yml file exists
- The workflow triggers on pull_request and on push to main
- CI runs in this order: install → typecheck → tests (unit + api + components) → build
- All three Vitest projects (unit, api, components) are enforced in CI
- A failing test, failing typecheck, or failing build causes the workflow to fail
- E2E (Playwright) is explicitly excluded from required CI, with a clear comment explaining why
- Any required GitHub secrets or env vars are clearly documented in the workflow file comments and in TESTING.md
- The workflow is fast enough to be practical for normal PR use
- The workflow is simple, readable, and maintainable by a future engineer

Out of scope
- CD or deployment orchestration
- Preview environment automation
- Infrastructure provisioning
- Playwright E2E in required CI
- Release tagging or version automation
- Rewriting tests or application code beyond minimal CI-compatibility fixes

CI Quality Rules (CRITICAL)
- Use the simplest implementation that satisfies the requirements
- Do not add unnecessary jobs, matrices, or abstractions
- Fail loudly on real errors
- Do not add flaky or unstable checks
- Do not invent commands that do not match the repo
- Prefer explicit readable steps over cleverness
- Keep runtime practical for everyday PR use

Tasks

1. Inspect and confirm the actual CI commands
Verify the exact commands to use for each stage:
- install: npm ci
- typecheck: npm run check
- tests: npx vitest run
- build: npm run build

Confirm the Node.js version from:
- .nvmrc
- .node-version
- engines field in package.json
Default to Node 20.x only if the repo does not specify a version.

Also inspect whether the api Vitest project requires a live PostgreSQL service in CI or whether storage/database access is already mocked sufficiently. vitest.config.ts may set DATABASE_URL to a test value, but confirm whether a real connection is actually needed before adding a service container.

2. Create .github/workflows/ci.yml
Create a single CI workflow that:
- triggers on pull_request
- triggers on push to main
- uses the correct Node version
- caches npm dependencies keyed on package-lock.json
- runs the pipeline in this exact order:
  1. install
  2. typecheck
  3. tests
  4. build

Use a clear, readable workflow structure.
Do not add extra jobs unless there is a real reason.

If API tests truly require PostgreSQL:
- add a postgres service container
- configure it minimally and correctly
- document why it is needed

If API tests do not require PostgreSQL:
- do not add the service
- document the decision clearly in comments or TESTING.md

Add a clear workflow comment explaining why Playwright E2E is excluded from required CI:
- it requires a running server and more setup/runtime
- it is intentionally excluded from the required fast PR workflow
- it can be added later as a separate optional or scheduled workflow if needed

3. Handle environment variables safely
Inspect whether npm run build or the test suite requires environment variables such as:
- OPENAI_API_KEY
- STRIPE_SECRET_KEY
- SESSION_SECRET
- DATABASE_URL
- any other app-specific config

Rules:
- Do not hardcode real secrets
- Use safe placeholder values only where appropriate and non-sensitive
- Do not invent fake production secrets unless required for the build/test to execute
- If secrets are required in GitHub Actions, document them clearly
- Prefer keeping CI secret-light where possible

4. Keep CI aligned with Task #88
This workflow must enforce the actual existing test foundation from Task #88:
- unit tests
- API integration tests
- component tests

Do not duplicate test logic unnecessarily.
Use the repo’s existing commands and structure.

5. Make only minimal CI-compatibility fixes
If small changes are needed to make CI work, make only minimal, targeted fixes.

Examples:
- command wiring
- env guards
- alias/path fixes that only fail in CI
- stable defaults for test/build

Do not use this task as an excuse to refactor unrelated app code.

6. Update TESTING.md
Add a concise CI section to TESTING.md documenting:
- what the workflow runs
- when it triggers
- what must pass for a PR to be accepted
- any required GitHub secrets or env vars
- why E2E is excluded from required CI

Do not create new documentation files.

Refactor Guardrail
- Do not refactor production code unless necessary to make CI work
- Any change must be minimal, isolated, and behavior-preserving
- If a bug is discovered, fix it and document it clearly in the final report

Relevant files
- package.json
- vitest.config.ts
- playwright.config.ts
- TESTING.md
- script/build.ts
- server/db.ts
- tests/setup.server.ts

Deliverables (REQUIRED)
When finished, provide:
1. the workflow file path
2. the exact triggers used
3. the Node version used and where it came from
4. the exact commands run in CI
5. whether PostgreSQL service was added or intentionally skipped, and why
6. any env vars or GitHub secrets required
7. all files created/modified
8. any CI-specific issues discovered and how they were resolved

Success criteria
- Every PR automatically runs validation
- Regressions are caught before merge
- The workflow matches the actual repo structure and commands
- The CI pipeline is simple, reliable, and maintainable
