Implement a bounded self-healing error recovery system in the existing Odigos codebase. This is an implementation task, not a planning task. Make concrete code changes.

Objective
Improve reliability by making transient failures self-heal automatically, while preventing retry storms, duplicate side effects, and endless loops.

Core principles
- Retry only errors that are likely transient
- Never retry client/validation errors that will not improve on retry
- Keep retries small and bounded
- Use exponential backoff with jitter
- Add a circuit breaker around the AI provider so repeated upstream failures fail fast
- Replay dead-lettered warehouse writes safely on startup / interval
- Ensure anything retried or replayed is idempotent or deduplicated
- Do not create infinite retry loops anywhere

Requirements

1) Shared error classification utilities
Create a small shared reliability/error-classification layer that distinguishes:
- retriable transient errors:
  - network failures / fetch failures
  - timeouts
  - 408
  - 429
  - 500
  - 502
  - 503
  - 504
- non-retriable errors:
  - 400
  - 401
  - 403
  - 404
  - validation/schema/input errors
  - unsupported file types / malformed requests
- side-effect-sensitive operations:
  - anything that could create duplicate user-visible outcomes must not be blindly retried unless idempotent

Done looks like
- A reusable helper exists for “isRetriableError”
- A reusable helper exists for jittered exponential backoff
- Retry decisions are centralized rather than ad hoc string matching scattered everywhere

2) Frontend: smart retry only for safe data fetching
Update React Query usage/config so safe read/query operations retry automatically, but mutations do not blindly retry.

Requirements
- For safe query fetches only:
  - retry max 2 times
  - exponential backoff with jitter
  - retry only on transient errors (network / 408 / 429 / 5xx)
  - do not retry 4xx validation/client errors
- Do not enable automatic retry for unsafe mutations like analyze or checkout unless already provably idempotent
- Preserve the existing anti-hammering intent: no aggressive infinite or high-count retrying

Done looks like
- Read/query endpoints self-heal from brief network/server blips
- Analyze/checkout are not silently re-submitted by the browser
- No infinite frontend retry loops exist

3) Backend: bounded retry wrapper for AI analyze path
Add a targeted retry wrapper around the upstream AI call used by the analyze route. Retry only the narrow AI provider call, not the full route logic.

Requirements
- Retry only on transient upstream AI failures:
  - timeout
  - connection reset/network failure
  - 429
  - 500/502/503/504
- Do not retry on:
  - invalid request payloads
  - prompt too large
  - auth/configuration errors
  - parsing/validation issues that indicate a non-transient problem
- Max attempts:
  - initial try + up to 2 retries max
- Use exponential backoff with jitter
- Add a per-attempt timeout
- Respect a total route/request time budget so retries do not make requests hang excessively
- Log retry attempts with structured context, but never log raw dealer text / uploaded document contents / PII

Done looks like
- Brief upstream AI failures recover automatically
- Analyze does not repeatedly retry forever
- The whole route is not duplicated / rerun unnecessarily
- Failures still return a clean 5xx/502-style response when exhausted

4) Circuit breaker around AI provider
Implement a lightweight in-process circuit breaker for the AI provider client.

Requirements
- Track recent AI call outcomes in a rolling window
- Count only dependency health failures toward the breaker:
  - timeout
  - network failure
  - 429
  - 5xx
- Do not count user/input/validation/parsing bugs toward opening the breaker
- Circuit states:
  - closed
  - open
  - half-open
- Open when failure threshold is exceeded within the rolling window
- While open, fail fast without continuing to hammer the provider
- After a cooldown, transition to half-open and allow a very small number of probe calls
- Close again if probes succeed
- Re-open if probes fail

Done looks like
- Repeated AI provider failure causes fast-fail behavior instead of repeated timeout storms
- Circuit breaker state transitions are visible in logs/metrics
- No endless half-open loop or unlimited probe traffic exists

5) Dead-letter queue replay for warehouse writes
The app already dead-letters failed warehouse writes. Add a bounded replay mechanism.

Requirements
- Add a replay worker/job that:
  - runs at startup and on a modest interval
  - fetches only eligible DLQ rows
  - leases/locks rows before processing so the same row is not replayed concurrently
  - re-attempts the warehouse write
  - marks success as resolved/removed
  - on failure, increments attempts and schedules next_attempt_at with exponential backoff
- Add/ensure DLQ fields needed for safe replay, such as:
  - attempt_count
  - max_attempts
  - next_attempt_at
  - status (pending / leased / resolved / dead or equivalent)
  - first_failed_at
  - last_failed_at
  - lease_expires_at or equivalent protection
  - last_error_code / last_error_message
- After max attempts, mark the item terminal/dead and stop retrying
- Never create an immortal replay loop
- Replay should be non-fatal to app startup if some items fail again
- Do not load the entire DLQ into memory at once

Done looks like
- Failed warehouse writes are retried later automatically
- Replay is bounded and idempotent/deduplicated
- Stuck rows eventually go terminal instead of looping forever

6) Idempotency / dedupe protections
Anything newly retried or replayed must be safe against duplication.

Requirements
- Use existing unique keys / upserts where available
- If needed, add a deterministic dedupe/idempotency key for replayed warehouse writes
- Ensure replaying the same failed write multiple times does not create duplicate warehouse records
- Do not weaken existing payment/checkout safety semantics

Done looks like
- Retries/replays do not create duplicate durable records
- The implementation clearly enforces at-most-once or safe-upsert semantics for replayed writes

7) Error boundary / recovery UX
Improve client recovery behavior so transient failures can recover without a full page reload.

Requirements
- For query-driven failures, allow reset/retry/refetch from the error UI
- Prefer retrying the affected fetch / resetting the component tree before telling the user to reload the whole page
- Keep behavior bounded; no auto-looping render retries
- Preserve user-entered form/input state where practical on failed operations

Done looks like
- Users can recover from transient UI/data failures with an in-app retry
- The app does not bounce between error and retry infinitely

8) Observability
Add targeted logging/metrics for the new healing mechanisms.

Requirements
Instrument at least:
- AI retry attempt count
- AI retry success-after-retry
- AI retry exhaustion/failure
- Circuit breaker open / half-open / close transitions
- DLQ replay attempt count
- DLQ replay success count
- DLQ terminal/dead count
- Optional but preferred: route latency impact when retries occur

Constraints
- Never log raw dealer submission text, uploaded document content, or PII
- Use existing observability patterns where possible (Sentry / metrics / technical reporting)
- Missing observability config must not crash the app

9) Guardrails / anti-looping rules
These are mandatory:
- Never retry indefinitely
- Never replay indefinitely
- Never retry 4xx client/validation errors except explicitly transient ones like 408/429
- Never aggressively retry the same operation in both frontend and backend in a way that multiplies requests uncontrollably
- Never retry unsafe side effects unless they are idempotent/deduplicated
- Never block startup just because DLQ replay has failures
- Never create setInterval / scheduler behavior that can overlap endlessly without lease/guard protection
