Implement Phase 2 of the UI migration for Odigos.

This phase is a tightly scoped landing-page presentation transplant only.

## PRIMARY GOAL

Use the new v0/Vercel homepage design as the visual reference/source and migrate only the safe, presentational sections of the existing Replit landing page so the homepage looks substantially closer to the new design without changing any product behavior.

This is a frontend presentation migration inside the existing Replit app.
Do NOT create a separate frontend/backend architecture.
Do NOT import the Vercel deployment as an app.
Do NOT re-platform anything.

## CRITICAL CONSTRAINTS

Do NOT:
- change analyzer logic
- change upload flow
- change OCR/text extraction flow
- change GPT/OpenAI flow
- change Stripe/payment flow
- change routes
- change API contracts
- change backend files
- change page behavior
- change CTA destinations unless the current app already uses those destinations
- change any existing forms or submission handlers
- change copy outside the sections being visually migrated from the new UI source
- do a global CSS/token rewrite
- refactor unrelated files
- touch any page other than the landing page and directly related presentational components

## FILE SAFETY

Use the migration baseline commit and migration-map.md as the source of truth for what is safe.
Only work in safe UI replacement zones already identified for the landing page.

## IMPLEMENTATION SCOPE

Migrate the landing page presentation in this order:

1. Header / top navigation
2. Hero section
3. Trust/supporting strip or equivalent social-proof section
4. Main marketing/value-prop sections
5. Sample output / example review section
6. Pricing section
7. FAQ section
8. Footer

## IMPORTANT IMPLEMENTATION RULES

- Preserve all existing landing-page behavior
- Preserve current links and route destinations unless the v0 section maps cleanly to an already-existing destination in this app
- Preserve accessibility and semantic HTML
- Reuse existing app structure where possible
- If the v0 source uses components incompatible with this app, adapt the markup/styling to the existing component system rather than forcing a broad component refactor
- Prefer localized styling/class changes over global styling changes
- If needed, create new presentational components under `client/src/migration/` or a clearly scoped landing-page component location, but do not use them anywhere except the landing page
- Do not import unused v0 abstractions that add complexity without benefit

## STRONGLY PREFER

- section-by-section replacement
- minimal blast radius
- visual parity without architecture churn
- preserving the current working routes and flows
- keeping diffs readable

## OUT OF SCOPE

- analyzer page redesign
- upload UI redesign
- shared design-system overhaul
- app-wide token/theme replacement
- checkout page/payment changes
- SEO/article page redesign
- backend cleanup/refactor

## SOURCE OF TRUTH

The v0/Vercel homepage is the visual/design source of truth.
The existing Replit codebase is the application/behavior source of truth.

If there is any conflict:
- preserve Replit behavior
- adapt the v0 design to fit the existing app safely

## DELIVERABLES

When complete, report back with:

1. exact files modified/created
2. which landing-page sections were migrated
3. confirmation of what behavior was intentionally preserved
4. any places where the v0 design had to be adapted to fit the current app
5. verification that no backend, route, analyzer, upload, OCR, or Stripe files were changed
6. confirmation that the landing page still loads cleanly with no new console or TypeScript errors

## REVIEW STANDARD

Do not claim success unless:
- the landing page visually moves meaningfully toward the v0 design
- no product behavior changed
- no non-landing-page flows were touched
