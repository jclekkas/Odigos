Task #111 — Wire feedback loop into scoring

Goal:
Use existing thumbs up/down data in `deal_feedback` to improve future analyses and expose agreement metrics in admin BI.

Required changes

1. Warehouse / dealer stats
   File: `server/warehouse/setupViews.ts`

Extend the core dealer aggregate so each dealer includes:

* `positive_feedback_count`
* `total_feedback_count`
* `feedback_agreement_pct`

Join path:

* `deal_feedback`
* joined to `dealer_submissions` via `listing_id`
* then grouped at dealer level the same way the current dealer aggregates are grouped

Rules:

* Positive feedback = thumbs-up / agreement
* Total feedback = all explicit thumbs-up/down ratings
* `feedback_agreement_pct = positive_feedback_count / total_feedback_count`
* guard divide-by-zero with `NULL` when no feedback exists
* keep the metric numeric in the warehouse layer; format later in app code

Important:

* Do not count missing / null feedback rows
* Keep the existing dealer aggregates intact
* Refresh / rebuild logic for the materialized view should still work cleanly

2. Market context
   Files:

* `server/marketContext.ts`
* `shared/schema.ts`

Update the market context types and fetch logic so dealer context can include:

* `feedbackAgreementPct?: number`
* `feedbackCount?: number`

Behavior:

* pull the fields from the dealer aggregate/view
* only include them in returned market context when `feedbackCount >= 3`
* follow the same thresholding pattern already used for `dealerAnalysisCount >= 3`
* do not fail the entire market-context fetch if feedback fields are missing; treat as optional

3. LLM prompt injection
   File: `server/routes/analyze.ts`

Extend the `MARKET_INTELLIGENCE` block so when dealer feedback signal is present it adds a sentence like:

“Users agreed with 62% of past 13 analyses for this dealer. Treat this as a calibration signal: if the case is borderline, express lower confidence rather than overstating certainty.”

Requirements:

* only include this line when `feedbackCount >= 3`
* keep existing market intelligence fields unchanged
* this is a confidence-calibration signal, not a hard override
* wording should push the model to be more cautious on borderline verdicts when agreement history is weak
* do not instruct the LLM to blindly follow prior users

Also:

* bump default `analysis_version` from `"v1"` to `"v2"` wherever the default is set for new analyses

4. Admin API
   New endpoint:

* `GET /api/admin/feedback-accuracy`

Protect it the same way as existing admin endpoints.

Response shape:

```ts
{
  overallAgreementRate: number | null,
  byScoreColor: {
    GREEN: number | null,
    YELLOW: number | null,
    RED: number | null
  },
  topDealers: Array<{
    dealerName: string,
    totalFeedbackCount: number,
    positiveFeedbackCount: number,
    agreementRate: number | null
  }>
}
```

Definitions:

* overall agreement rate = all positive feedback / all explicit feedback
* by-score-color = agreement rate for analyses grouped by final score color
* topDealers = top 10 dealers by total feedback volume, descending
* only include dealers with `totalFeedbackCount >= 3`
* if multiple dealer-name fields exist in warehouse/app code, use the same canonical dealer name already used elsewhere in BI

Implementation notes:

* prefer warehouse/query-side aggregation, not loading large raw datasets into memory
* keep null-safe handling for empty datasets
* preserve existing admin auth pattern exactly

5. Admin dashboard UI
   File: `client/src/pages/admin-business.tsx`

Add a new “Feedback Accuracy” section that calls `/api/admin/feedback-accuracy`.

UI should show:

* overall agreement rate
* agreement rate cards for GREEN / YELLOW / RED
* table of top 10 dealers by feedback volume with:

  * dealer name
  * total feedback
  * positive feedback
  * agreement %

Guidelines:

* follow the styling/patterns already used in admin BI
* show percentages as rounded whole numbers unless the page already uses 1 decimal place consistently
* include loading and error states
* empty-state copy should be sensible when there are not enough feedback records yet

Suggested implementation details

Warehouse SQL sketch:

```sql
SUM(CASE WHEN df.feedback = 'up' THEN 1 ELSE 0 END) AS positive_feedback_count,
COUNT(CASE WHEN df.feedback IN ('up','down') THEN 1 END) AS total_feedback_count,
CASE
  WHEN COUNT(CASE WHEN df.feedback IN ('up','down') THEN 1 END) > 0
  THEN SUM(CASE WHEN df.feedback = 'up' THEN 1 ELSE 0 END)::float
       / COUNT(CASE WHEN df.feedback IN ('up','down') THEN 1 END)
  ELSE NULL
END AS feedback_agreement_pct
```

Prompt logic sketch:

* if `feedbackCount >= 3` and `feedbackAgreementPct` is finite:

  * compute whole-number percent
  * append one extra line to `MARKET_INTELLIGENCE`

Example injected line:

```txt
Users agreed with 62% of past 13 analyses for this dealer. Use this as a confidence-calibration signal: for borderline cases, avoid overstating certainty.
```

Admin endpoint query expectations

* overall: one aggregate query
* by color: grouped aggregate by final score color
* top dealers: grouped aggregate ordered by feedback volume desc limit 10

Acceptance criteria

* dealer feedback fields exist in warehouse dealer aggregate
* `getMarketContext()` returns feedback signal only when threshold met
* analyze prompt includes dealer feedback sentence only when available
* new analyses default to `analysis_version = "v2"`
* `/api/admin/feedback-accuracy` returns valid aggregates
* admin BI page renders the new section cleanly
* app builds and runs without breaking existing analysis flow

Non-goals

* no model fine-tuning
* no public display of feedback
* no per-user personalization
* no automatic score overrides based on feedback history
